# Effort Estimator — 工数估算技能

## 核心原则

### 不做无脑计算器

这个 skill 不是一个「输入需求 → 输出数字」的机械工具。它是一个**掌握多种方法论的估算顾问**。

每次被调用的工作流：
1. **诊断** — 了解项目阶段、需求明确度、团队情况
2. **推荐** — 从方法库中选出最合适的方法（可以组合）
3. **分解** — 用选定的方法进行 WBS 或参数推算
4. **校准** — 加入技能系数、管理工数、风险缓冲
5. **输出** — 根据场景选择合适的输出格式

### 什么时候用哪种方法

选择方法不看个人偏好，看这三个维度：

| 维度 | 判断标准 |
|------|---------|
| **需求明确度** | 有详细要件书？只有一句话？介于两者之间？ |
| **历史数据** | 团队有类似项目实绩？有行业基准？完全新领域？ |
| **精度要求** | 概算（±50%）？预算申请（±30%）？契约見積（±10%）？ |

大致推荐逻辑（详见 `references/methods.md`）：

- 需求模糊 + 无历史 → **类比法**（快速拍范围）+ **三点法**（覆盖不确定性）
- 需求明确 + 有历史 → **WBS 自下而上**（精度最高）+ 参数法辅助校验
- 固定预算 + 可调范围 → **プライスツーウィン法**（先定预算再裁剪功能）
- 大项目 + 团队成熟 → **標準タスク法** + **FP法**做交叉验证
- 改修/保守 → **プログラムステップ法** 辅助，WBS 为主

## 工作流

### Phase 1: 诊断

遇到估算需求时，先了解这些——**关键在于分轮次、按优先级，不要一次抛太多问题**：

**第一轮（必问，3-4个）**：项目类型 + 阶段 + 需求明确度 + 估算目的
**第二轮（视情况，2-3个）**：团队情况 + 客户约束 + 历史参考
**日本案件追加（1-2个）**：契約形態 + 間接工数の明示有無

⚠️ **追问原则**：
- 已有信息不要重复问。如果信息足够（如要件书），直接进入估算，只问最关键的缺口
- 信息不足时**先问清楚再估，不凭空猜测**。缺少的信息往往直接决定估算范围
- 问了关键问题后，**同时给出基于现有信息的宽范围暂定估算**——标注「待确认后缩窄」。不要让用户空手而归

#### 必须弄清的项目基础

1. **项目类型**：新规开发 / 改修 / PoC / 保守运维 / AI·RAG案件 / 其他
2. **项目阶段**：刚拿到一句话需求 / 有了要件书 / 设计已定 / 正准备报价
3. **技术栈大致方向**：Web（前后端+框架）/ Mobile / AI·LLM / 連携（Kintone等）/ 基础设施
4. **需求明确度**：
   - ◎ 有详细要件书，功能列表清晰
   - ○ 有大方向但细节未定
   - △ 只有一个想法或一句话需求
5. **团队情况**：
   - 单人全栈 / 小团队(2-5人) / 中型(5-15人)
   - 成员级别：初级多还是高级多
   - 是否有该领域的开发经验
6. **有没有历史参考**：做过类似项目吗？实际花的人日知道吗？
7. **客户的约束**：
   - 有固定预算吗？
   - 截止日期是否已定？
   - 有没有已经决定的供应商/技术/平台？
8. **估算目的**：
   - 内部摸底 / 给客户报预算范围 / 正式契约見積 / 申请预算

#### 日本案件专属追加问题

当识别到日本案件时（日语需求书、日本客户、Kintone/Japan特定技术），额外了解：

- 是否包含 **間接工数**（定例会議、稟議資料、進捗報告書）的明示要求
- 客户是 **請負契約** 还是 **準委任契約**（影响工数责任边界）
- 是否需要 **相見積もり**（多方报价对比）
- 月額保守費用是否需要单独列示

### Phase 2: 方法推荐

根据诊断结果，用下面的逻辑推荐方法：

```
if 需求明确度 == "◎" and 有历史数据:
    主 = WBS自下而上, 辅 = 参数法/FP法校验
elif 需求明确度 == "○" or (需求明确度 == "◎" and 无历史):
    主 = WBS + 三点法/PERT（每个task用三点估算）, 辅 = 类比法对照
elif 需求明确度 == "△":
    if 有经验丰富的专家团队:
        主 = 德尔菲法 + 三点法, 辅 = NESMA快速FP法概算 + 类比法
    else:
        主 = 类比法（出范围）+ 三点法（算期望值）, 辅 = NESMA快速FP法概算
elif 预算固定:
    主 = プライスツーウィン（根据预算裁剪） + WBS

# 团队参与的估算（推荐附加）
if 团队人数 >= 2 and 任务不确定性高:
    推荐使用 Planning Poker 或宽带德尔菲法 → 炸出认知偏差、达成共识
```

**⚠️ 重要**：
- 推荐方法后，向用户确认「这个方法适合吗？还是你有偏好的方法？」再往下走
- **不要让一个人估自己要做的事** — 尽量拉多人交叉验证
- **「拍脑袋」未必不准** — 有经验的专家直觉估算有时和形式化方法一样可靠（相关系数可达 0.89），关键在于执行者的经验水平

### Phase 3: 分解与计算

详见 `references/methods.md`，各方法的完整公式和操作步骤。

**⚠️ 开始计算前，先回答这 4 个问题（防 buffer 叠加）**：

1. 本次估算的主方法是什么？（WBS单值 / WBS+三点法 / 三维评估 / 其它）
2. 如果用了三点法(PERT)——每个 task 已含乐观/悲观区间，**最后统一缓冲要减半**
3. 如果用了三维评估——B系数已经内嵌缓冲，**不再另加统一风险缓冲**
4. 最终缓冲率是多少？**如果超过 50%，回头检查**看哪里叠加了

核心步骤：
1. **WBS 分解** — 按 8/80 原则（最少 1 人日，最多 2 周），拆到可单独估算的粒度
2. **纯工数计算** — 用选定方法算出每个 task 的纯工作时间
3. **技能系数校准** — 根据分配人员的级别乘系数：
   - Senior: ×0.8~1.0
   - Mid: ×1.2~1.5
   - Junior: ×2.0~3.0
   - 单人全栈的情况：按你的自我评估设基准系数 1.0
4. **管理工数加成** — 纯工数基础上加 10%~20%（会议、文档、沟通、レビュー）
5. **风险缓冲**：
   - 熟悉领域：+10%~15%
   - 第三方連携/新技术：+30%~50%
   - 需求模糊：+50%~100%（用三点法覆盖更好）

⚠️ **关键：buffer 不能重复叠加（anti-double-buffer 规则）**：
- **三点法(PERT)本身已含不确定性** — 如果用 PERT 计算了每个 task 的期望值，最终风险缓冲**减半**
- **三维评估模型本身已含缓冲系数 B** — 如果用了 B 系数调整每个 task，不再加统一风险缓冲
- **两种方法都用** — 只取**较大的那个缓冲值**，不要相加
- 简单检查：最终的缓冲比例如果超过 50%，回头检查是否 double-counting 了

#### AI·LLM 案件专属加成

AI/RAG案件额外考虑：
- 模型选型和 prompt 调优的迭代成本（通常 3-5 轮）
- Embedding/RAG 的精度调优
- 音声認識的精度测试和用语辞書注入
- 这些属于高不确定性工作，建议每个此类 task 用三点法独立估算

### Phase 4: 输出

根据场景选择格式，详见 `references/output-templates.md`：

| 场景 | 推荐输出格式 |
|------|------------|
| 客户正式見積 | 日式見積書（工程×工数×単価 + 諸経費 + 月額） |
| 内部立项 | WBS 详表 + PERT 三点期望值 |
| 快速摸底 | 一頁摘要（按模块的大致范围） |
| 预算申请 | 按 phase 拆分 + 风险缓冲明示 |
| 多人团队 | 人员配置表 + スキル係数付き工数 |

输出时 **必须解释**：
- 用了什么方法、为什么选它
- 关键假设是什么（如果假设变了数字会怎么变）
- 哪些部分不确定性高，建议什么时候重新估算
