# 估算方法论详解

## 一、方法总览与选择矩阵

### 什么时候用哪种方法

| 方法 | 精度 | 耗时 | 需求明确度要求 | 历史数据要求 |
|------|------|------|--------------|------------|
| 自下而上 (Bottom-up/WBS) | ★★★★★ | 高 | ◎ 必须有详细要件 | 不必须 |
| 三点法/PERT | ★★★★☆ | 中 | ○ 有大方向即可 | 不必须 |
| 类比法 (Analogous) | ★★☆☆☆ | 低 | △ 一句话即可 | 必须有 |
| 参数法/FP法 | ★★★★☆ | 中 | ○ 需要功能清单 | 必须有 |
| 德尔菲/宽带德尔菲 | ★★★★☆ | 高 | △ 任何阶段 | 不必须 |
| 三维评估模型 | ★★★★☆ | 中 | ○ 有功能列表即可 | 部分需要 |
| 標準タスク法 (日式) | ★★★★☆ | 中 | ○ 需要有 WBS 模板 | 必须有模板 |
| プライスツーウィン (日式) | ★★★☆☆ | 低 | △ 一句话即可 | 不必须 |
| NESMA快速FP法 | ★★★☆☆ | 低 | △ 可研阶段 | 必须有生产率基准 |

### 方法选择决策树

```
项目阶段?
├── 初期（一句话需求）
│   ├── 有类似项目实绩 → 类比法 + 30~50%缓冲
│   └── 无实绩 → 三点法 (宽范围) + 功能点粗略估算
├── 中期（有要件书但细节未定）
│   ├── 预算已定 → プライスツーウィン (按预算裁剪功能)
│   └── 预算未定 → WBS分组块 + 三点法 + 类比法交叉验证
└── 后期（要件确定）
    ├── 团队成熟 → WBS自下而上 + FP法校验
    └── 团队新手 → WBS + 三点法 (降低个人偏差) + 技能系数校准
```

## 二、各方法详细操作步骤

### 2.1 自下而上法 (Bottom-up Estimation / WBS估算法)

**核心思想**：拆到最小单元，逐项估算，层层加总。

**操作步骤**：

1. **创建 WBS (Work Breakdown Structure)**
   - 按功能模块→子功能→具体任务 三级分解
   - 遵循 8/80 原则：每个底层任务 ≥ 1人日（8小时），≤ 2周（80小时）
   - 每个任务必须是可交付、可验证的独立单元

2. **逐项估算纯工数**
   - 对每个底层任务问：一个熟练开发者需要多少纯工作时间？
   - 只算"写代码/做设计"的时间，不含会议、沟通
   - 单位：人时（人小时）或人日（1人日=8小时）

3. **技能系数校准**
   - 纯工数是按"标准熟练度"算的
   - 根据实际分配人员的级别乘系数：
     | 级别 | 系数 | 说明 |
     |------|------|------|
     | Senior/高级 | 0.8~1.0 | 熟悉业务+技术的资深者 |
     | Mid/中级 | 1.2~1.5 | 能独立工作但需偶尔指导 |
     | Junior/初级 | 2.0~3.0 | 需要较多指导和检查 |

4. **加入管理工数 (Management Overhead)**
   - 定例会議、日报周报、代码审查、文档编写、跨部门沟通
   - 通常加 10%~20% 到纯工数上
   - 日本案件要特别注意：稟議資料作成、進捗報告会、お客様定例 等

5. **加入风险缓冲**
   - 熟悉领域 + 稳定需求：+10%~15%
   - 第三方API連携/未経験技術：+30%~50%
   - 需求模糊或客户变更频繁：+50%~100%

6. **汇总公式**：
   ```
   最終工数 = Σ(各Task純工数 × 技能係数) × (1 + 管理工数率) × (1 + リスク係数)
   ```

**WBS 示例（日报模块）**：

```
日报モジュール
├── 1.1 ドラフト箱機能 (3人日)
│   ├── 1.1.1 学校検索・選択UI (1人日)
│   ├── 1.1.2 採用履歴表示 (kintone連携) (1人日)
│   └── 1.1.3 ヒアリング項目表示 (0.5人日)
│   └── 1.1.4 推奨事項表示 (pending_items連携) (0.5人日)
├── 1.2 音声入力・チャット機能 (5人日)
│   ├── 1.2.1 音声認識UI (2人日)
│   ├── 1.2.2 AIチャット対話エンジン (2人日)
│   └── 1.2.3 文字起こし→構造化変換 (1人日)
├── 1.3 確定処理 (2人日)
│   ├── 1.3.1 kintone書込連携 (1人日)
│   ├── 1.3.2 Embedding・RAG保存 (0.5人日)
│   └── 1.3.3 pending_items抽出・更新 (0.5人日)
├── 1.4 採点・フィードバック (2人日)
│   ├── 1.4.1 採点ロジック (1人日)
│   └── 1.4.2 改善アドバイス生成 (1人日)
└── 1.5 kintoneマスタ連携 (2人日)
    ├── 1.5.1 学校台帳API (0.5人日)
    ├── 1.5.2 教員マスタAPI (0.5人日)
    ├── 1.5.3 ヒアリング項目API (0.5人日)
    └── 1.5.4 ナレッジAPI (0.5人日)

純工数合計: 14人日
仮定: 全栈1人 (系数1.0) → 14人日
管理工数 15%: 14 × 1.15 = 16.1人日
风险缓冲 30% (kintone連携+AIあり): 16.1 × 1.30 = 20.9人日
→ 約21人日
```

### 2.2 三点估算法 (Three-Point Estimation / PERT)

**核心思想**：不赌一个数字，用三个值算期望，覆盖不确定性。

**公式**：
```
期待値 E = (O + 4M + P) / 6
標準偏差 σ = (P - O) / 6
```

其中：
- **O (Optimistic)**：一切顺利的最短时间
- **M (Most Likely)**：正常情况的最可能时间
- **P (Pessimistic)**：最坏情况的最长时间

**操作步骤**：

1. 每个 task 设定三个值，分别估计 O / M / P
2. 用公式算出期望值 E
3. 算标准差 σ，评估不确定性
4. 可选项：用 Monte Carlo 思维——P±σ 范围内的概率约 68%，P±2σ 约 95%

**三点法示例（音声認識機能）**：

```
Task: 音声認識UI + AIチャット対話エンジン
- O (乐观): 3人日 (APIがスムーズ、UIライブラリが使える)
- M (最可能): 5人日 (普通のトラブル込み)
- P (悲观): 10人日 (音声精度問題、用語辞書チューニング難航)

E = (3 + 4×5 + 10) / 6 = 33/6 = 5.5人日
σ = (10 - 3) / 6 = 1.17人日

→ 区间: 4.3〜6.7人日 (68%置信), 3.2〜7.8人日 (95%置信)
```

**做汇总 PERT 时**：
- 不要只加总 E值，要把所有task的σ也考虑进去
- 汇总 σ_total = √(σ₁² + σ₂² + ... + σₙ²)
- 最终报价可以用 E_total + σ_total（68% 概率不超）或 E_total + 2σ_total（95% 概率不超）

### 2.3 类比法 (Analogous Estimation)

**核心思想**：找最相似的过去项目，按差异调整系数。

**操作步骤**：

1. 从历史项目库中找到最相似的 2-3 个项目
2. 对比差异维度：
   - 功能规模（画面数 × API数 × DB表数）
   - 技术复杂度（連携先有无、AI有无、性能要求）
   - 团队差异（人数、技能水平）
3. 对每个差异维度设调整系数（0.7~1.5）
4. 加权平均后得估算值
5. 必加较大缓冲（因为精度最低）：+30%~50%

**类比法示例**：

```
参考项目A: 某社内業務システム (新規開発)
- 規模: 画面6枚, API 12本, DB 6表
- 実工数: 25人日 (1人全栈)
- 技術: React + Node.js + PostgreSQL

現案件 (文英堂):
- 規模: 画面4枚, API 8本, DB 4表
- 規模比: (4+8+4)/(6+12+6) = 16/24 = 0.67
- 技術差: +AI(RAG+採点) → ×1.5, +kintone連携 → ×1.2
- 複合係数: 0.67 × 1.5 × 1.2 = 1.20

估算: 25 × 1.20 = 30人日
加40%缓冲 (初めてのkintone案件): 30 × 1.40 = 42人日
→ 40〜45人日 范围
```

### 2.4 功能点法 (Function Point Analysis / FP法)

**核心思想**：从用户视角数"功能"，不依赖技术实现。

**五类功能点**：

| 类型 | 含义 | 示例 |
|------|------|------|
| EI (外部入力) | 用户输入数据的操作 | 登录、注册、表单提交 |
| EO (外部出力) | 系统输出给用户 | 报表、导出CSV、通知 |
| EQ (外部照会) | 用户查询数据 | 检索画面、列表表示 |
| ILF (内部論理File) | 系统内部数据群 | 用户表、订单表、日报表 |
| EIF (外部接口File) | 引用外部数据 | kintone参照、外部API |

**步骤**：

1. 识别所有功能，按类型和复杂度（Low/Average/High）评分
2. 加权求和得 未調整FP
3. 乘以 VAF (Value Adjustment Factor，0.65~1.35，14个系统特性）
4. 得 調整後FP
5. 用团队的生产率（人日/FP）换算工数

**FP法的局限性**：
- 需要受过培训的人才能准确计数
- 对算法密集型/AI案件不敏感
- 建议作为 WBS 的交叉验证手段，不做为主估算方法

### 2.5 標準タスク法（日式）

日本IT業界特有の手法。IPA（情報処理推進機構）が整備。

**特徴**：
- あらかじめ定義された「標準タスク」一覧から該当タスクを選ぶ
- 各タスクに標準工数が設定されている
- WBSと組み合わせて使うことが多い

**主なタスクカテゴリ**：
- プロジェクトマネジメント
- 要件定義
- 基本設計
- 詳細設計
- 製造（プログラミング）
- 単体テスト
- 結合テスト
- 総合テスト
- 移行・導入

**メリット**：プロジェクト管理との親和性が高い、見積もり段階から綿密な計画を立てられる
**デメリット**：標準タスク定義がない分野には適用困難

---

### 2.6 プライスツーウィン法 (Price-to-Win)

予算が先に決まっている場合の手法。

**手順**：
1. クライアントの予算を確認
2. WBSで全機能をリストアップ
3. 各機能に優先順位をつける
4. 予算内に収まるように Must-have / Should-have / Nice-to-have に分類
5. Must-have だけ版の工数と全機能版の工数を両方提示
6. フェーズ分割提案（Phase 1: Must-have, Phase 2以降: 残り）

**注意点**：品質低下や機能削減のリスクがあるため、必ず「削ったもの」を明示すること。

## 三、調整係数の設定基準

### 3.1 技能係数（個人差）

実際の開発者の生産性差を数値化。同じタスクでも人によって2〜3倍の差が出る。

- **Senior（5年〜）**：0.8〜1.0 — 自律的に動ける、コードレビュー不要、アーキテクチャ判断可
- **Mid（2〜5年）**：1.2〜1.5 — 独り立ち可能だが設計判断にはシニア確認必要
- **Junior（〜2年）**：2.0〜3.0 — 詳細指示が必要、レビュー前提

**プロジェクト単位の算出式**：
```
チーム技能係数 = (Senior人数×1.0 + Mid人数×1.3 + Junior人数×2.5) / 総人数
```

### 3.2 管理工数率（共通）

開発以外の「間接工数」の割合。これを忘れると必ず遅延する。

| 作業種別 | 内容 | 参考比率 |
|---------|------|---------|
| 定例会議 | デイリー、週次、月次 | 5〜10% |
| ドキュメント作成 | 設計書、レビュー資料、議事録 | 5〜8% |
| 進捗報告 | 顧客向け報告書作成 | 3〜5% |
| コードレビュー | PR review | 3〜5% |
| 環境管理 | CI/CD整備、デプロイ | 2〜3% |
| **合計** | | **10〜20%** |

日本案件は特に「お客様定例」「稟議資料」などの間接工数が多くなりがちなので、20%側で見積もる。

### 3.3 リスク係数

| リスク要因 | 低リスク | 中リスク | 高リスク |
|-----------|---------|---------|---------|
| 技術の新規性 | 使い慣れた技術 | 一部新技術 | 全くの新技術 |
| 外部連携 | なし or 既知API | 1〜2システム | 3システム以上 or 未知API |
| 要件の安定性 | 確定済み | 大枠確定、詳細変更有 | 未確定多数 |
| チーム経験 | 類似案件3件以上 | 類似案件1〜2件 | 初めての領域 |
| 納期プレッシャー | 余裕あり | 普通 | 厳しい |

**総合リスク係数の目安**：
- 低リスク中心：+10%〜15%
- 中リスク中心：+20%〜35%
- 高リスク中心：+40%〜100%（この場合は三点法の方が適切）

### ⚠️ バッファ重複禁止ルール（anti-double-buffer）

**最もよくある間違い**：PERTで既に不確実性を織り込んだのに、さらに30%バッファを上乗せしてしまう。

| 状況 | 正しい処理 |
|------|----------|
| PERT三点法を使った | 最終リスクバッファは**半減**（PERTが既に幅を持っているため） |
| 三ディメンション評価のB係数を使った | **それ以上の統一バッファは不要**（B係数が既にリスク調整済み） |
| PERT + 三ディメンション両方使った | **大きい方のバッファ値だけ**適用、加算しない |
| 単一値估算 + バッファ無し | 通常通り10%~50%適用 |

**簡易チェック**：最終バッファ率が50%を超えたら、重複の可能性が高い。直ちに計算過程を見直すこと。

## 四、精度向上の実践テクニック

### 4.1 多方对齐（複数人レビュー）

PM だけが数字を出すのは危険。以下の役割でチェック：
- **実装者**が各タスクを見積もる（現場の肌感）
- **技術リード**が妥当性をレビューする（技術リスクの目利き）
- **PM**が集計し、過去実績との乖離をチェック

### 4.2 ベロシティ追跡（Velocity Tracking）

- 毎回のプロジェクトで「見積もり vs 実績」を記録
- 2〜3プロジェクト追跡すれば、チームの「見積もり癖」が見える
- 楽観しがちなチームか、逆にバッファを盛りすぎるチームかが分かる

### 4.3 パーキンソンの法則対策

「仕事は与えられた時間いっぱいまで膨張する」。
- バッファは PM が一元管理し、個別タスクに乗せない
- Critical Chain Method：個別タスクのバッファを削り、プロジェクト末尾にまとめて置く

### 4.4 段階的詳細化（Rolling Wave Planning）

- 直近 2 週間のタスク：1人日単位で精密に見積もる
- 1〜2ヶ月先のタスク：3〜5人日単位でラフに見積もる
- 3ヶ月以上先：モジュール単位の概算
- プロジェクト進行に合わせて段階的に詳細化する

## 五、德尔菲法 & 宽带德尔菲法

**核心思想**：专家独立估算 → 匿名汇总 → 讨论分歧 → 多轮迭代 → 趋近共识。避免权威压制和从众效应。

**标准德尔菲法（Delphi Method）步骤**：

1. 选择 5-10 名专家（开发/测试/架构/产品）
2. 第一轮：各专家**匿名、独立**估算每个任务
3. 收集所有估算值，统计中位数和四分位距
4. 将汇总结果（不含姓名）反馈给所有专家
5. 第二轮：专家看到群体意见后重新估算
6. 重复 2-4 轮直到意见收敛
7. 最终取中位数或平均值作为估算结果

**宽带德尔菲法（Wideband Delphi）**—— 德尔菲的改进版：
- 增加**面对面讨论**环节（每轮估算后）
- 最高和最低估算者必须解释理由
- Planning Poker 就是宽带德尔菲的变体

**Planning Poker（计划扑克）操作详解**：

- 使用斐波那契数列牌：0, 0.5, 1, 2, 3, 5, 8, 13, 20, 40, 100, ∞
- 单位：故事点或人天均可
- 步骤：
  1. PO 讲解用户故事
  2. 全员**独立选牌**（不同时公开）
  3. **同时亮牌**
  4. 最高和最低者解释理由
  5. 讨论后重新选牌，直到收敛
- **核心价值**：不为了精确数字，而是为了**炸出认知偏差**——"老张觉得简单"和"小李觉得很难"之间的 gap

## 六、三维评估模型（F-D-U 模型）

**来源**：综合多位一线 PM 的经验提炼。适用于**需求不完全确定但需要结构化估算**的场景。

**三个维度各打 1-5 分**：

| 维度 | 1分 | 2分 | 3分 | 4分 | 5分 |
|------|-----|-----|-----|-----|-----|
| **功能复杂度(F)** | 简单CRUD | 常规功能 | 并发/事务/幂等 | 算法/性能/多端 | 全新技术栈 |
| **系统依赖(D)** | 无外部依赖 | 内部稳定系统 | 1-2外部系统 | 多系统/接口不规范 | 遗留系统/无文档 |
| **认知不确定性(U)** | 需求清晰+历史案例 | 少量细节待定 | 核心清晰/边界模糊 | 文档缺失 | 范围易变/无参考 |

**计算公式**：
```
综合风险系数 R = (F + D + U) / 3
缓冲系数 B = 1 + (R - 3) × 0.3
推荐工时 = 基础工时 × B
```

**示例**：用户登录功能，F=3(并发/安全)，D=4(多系统认证)，U=5(安全合规要求多)
→ R=(3+4+5)/3=4，B=1+(4-3)×0.3=1.3，基础3人日→推荐3×1.3=3.9人日

**优点**：让每个估算有"可解释的根拠"，不再是无脑拍数字。效果类似给每个 task 做了 mini 风险审计。

## 七、NESMA 快速功能点法

**来源**：荷兰 NESMA 组织，IFPUG 的简化变体。适用于**可研阶段需求不完整时**快速估算规模。

**核心简化**：不再逐项判断复杂度，直接设定默认复杂度：
- ILF（内部逻辑文件）：直接计 **7 FP**（对应低复杂度）
- EIF（外部接口文件）：直接计 **5 FP**（对应低复杂度）
- EI（外部输入）：直接计 **4 FP**（对应中复杂度）
- EO（外部输出）：直接计 **5 FP**（对应中复杂度）
- EQ（外部查询）：直接计 **4 FP**（对应中复杂度）

**换算工数**：
```
开发工作量 = FP总数 × 生产率系数（如 1.36 人日/FP）
总工作量 = FP总数 × 总生产率系数（如 1.87 人日/FP）
```

**一个值得注意的发现**：曹博士对某中心 15 个历史项目的分析表明——传统"拍脑袋"估算结果与 NESMA 功能点规模的**相关系数达 0.89**（高度相关），说明有经验者的直觉估算未必不如形式化方法。关键在于**执行估算的人是否有足够经验**。

**适用建议**：用于可研阶段快速校验"拍脑袋"结果的合理性，不替代 WBS 详细估算。

## 八、敏捷估算工具箱

当程序员说「没做过，不知道要多久」时，PM 不是逼他们给数字，而是帮他们拆到能估的粒度。

### 8.1 拆到能估（核心原则）

> "不是问多久，而是拆到你肯定知道多久。"
> "只要能拆，就能评估。不能评估，只是因为还没拆。"

操作：
1. 把「不确定的整体」拆成「确定的小块」
2. 让程序员指出 80% 确定的部分（各自挂上时间）
3. 对 20% 不确定的部分——用 Spike 或 Timebox

### 8.2 Timebox（时间盒）

不给不确定的任务估算死线，而是限定**尝试时间**：
> "我们给这个未知模块 3 小时做最小验证。3 小时后决定继续还是换方案。"

### 8.3 Spike 探针任务

对黑盒技术（团队从未接触过），不估算功能本身，先估算**调研时间**：
- "老张，我不逼你现在估时。给你 1 天时间，就研究这个 SDK。明天告诉我这个功能到底要 3 天还是 13 天。"
- 本质：把「无限未知」变成「有限调研」

### 8.4 Story Point + Velocity

- **Story Point**：不是时间，是**相对复杂度**（含工作量+风险+不确定性）
- **Velocity**：团队一个 Sprint 能完成的 Story Point 数量
- **推算**：总 Story Point ÷ Velocity = 需要几个 Sprint
- 关键：Point 和 Velocity 需要 2-3 个 Sprint 的磨合才趋稳

### 8.5 Reference 借鉴对比

> "之前做过类似的吗？复杂度相似吗？"
程序员很难估完全未知的东西，但很擅长估「与旧功能对比」的东西。

## 九、人性维度——估算不是数学，是博弈

### 9.1 PM 的真实角色

PM 在估算中的本质角色不是「算数字的人」，而是**居间调停者**——在管理者（要快）和程序员（要稳）之间找到平衡。

### 9.2 信息不对称的三层猜疑链

- 程序员知道自己能力，但**没有动机做准确评估**——估准了没奖励，估少了背锅
- 于是默认**防御性估算**：心里默默乘 1.5~3 倍
- 管理者知道程序员多估，程序员知道管理者知道……
- PM 的 buffer 层层传递：程序员报 18 天 → PM 向管理层报 24 天 → 实际可能 6 天就够

### 9.3 应对策略

- **不要让自己估自己的活**——会有各种偏差
- **多方对齐**——实际执行者估 + 技术 Lead 审 + PM 汇总校准
- **匿名估后再讨论**——消除权威压制（德尔菲法的精髓）
- **估算是团队的承诺，不是 PM 的命令**——数字是团队自己定的，才有承诺感
- **buffer 单独列，PM 统一管**——不要揉进每个 task

### 9.4 防御性估算的三种根源

1. **幸存者偏差**：老张觉得简单（搭个 Redis 2 小时），新手实际要 3 天
2. **防御性估算**：怕背锅，3 天的活报 9 天
3. **认知不一致**：产品想的「登录」≠ 开发想的「登录」（含密码找回/第三方/风控）

## 十、持续校正——让估算越来越准

### 10.1 双周校正机制

每两周开一次估算复盘会（开发+测试+PM）：
- 实际 vs 估算的差异
- 差异原因分类（需求变更/技术坑/低估/漏估）
- 更新团队的「偏差系数」

### 10.2 工作量偏差看板

| 任务 | 估算 | 实际 | 偏差率 | 触发复盘 |
|------|------|------|--------|---------|
| 用户登录 | 8人日 | 14人日 | +75% | ✅ |
| 数据导出 | 5人日 | 6人日 | +20% | ❌ |

规则：偏差 > 30% 自动触发复盘。

### 10.3 失败推演机制

每个功能点列出 3 个「最可能失败场景」，自动预留应对工时：
- 场景1：外部接口超时 → 熔断/降级
- 场景2：大文件上传 → 流式处理
- 场景3：权限遗漏 → 安全审计

### 10.4 偏差归因 → 组织学习

项目结束后：
1. 列出所有「估 vs 实」偏差项
2. 每项归因到一个「认知盲点」
3. 写入**系统认知手册**，后续项目参考

### 10.5 建立历史工时台账

| 记录项 | 说明 |
|--------|------|
| 功能类型 | CRUD/連携/AI/报表 等 |
| 实际工时 | 含开发+测试+联调 |
| 负责人员级别 | Senior/Mid/Junior |
| 遇到的坑 | 技术/需求/沟通 |
| 下次同类估算系数 | 基于实际偏差调整 |

## 十一、国标参考：GB/T 36964-2018

中国国家标准《软件工程 软件开发成本度量规范》核心要点：

1. **先定规模**：用功能点法数清软件的「大小」
2. **找基准数据**：参考行业基准（如北京软件造价联盟年度数据）
3. **套公式**：`工作量 = 规模 × 生产率系数 ± 调整因子`
   - 调整因子：工期压力/技术难度/需求变更频率
4. **交叉验证**：至少用两种方法，结果差太多（>2倍）就回头重审需求
5. **输出范围值**：800-1000人日，不是「900人日」一个死数字
6. **直接人力 + 间接人力(15-30%) + 非人力成本**
