# 日本IT業界の見積もりコンテキスト

## 一、用語・単位体系

### 基本単位

| 日本語 | 中国語 | 定義 |
|--------|--------|------|
| 人時（にんじ） | 人时 | 1人×1時間 |
| 人日（にんにち） | 人日 | 1人×1日（実働8時間が標準） |
| 人月（にんげつ） | 人月 | 1人×1ヶ月（実働20日が標準、160時間） |

### 見積もり関連用語

| 日本語 | 意味 | 備考 |
|--------|------|------|
| 見積もり（みつもり） | 估算/报价 | 正式な契約前に提出する |
| 概算（がいさん） | 粗略估算 | 精度±50%程度、初期段階 |
| 準委任契約 | 准委任合同 | 工数ではなく「善管注意義務」ベース |
| 請負契約 | 承包合同 | 成果物ベース、工数超過は受託者負担 |
| どんぶり勘定 | 拍脑袋预算 | 大雑把な感覚的管理（戒められる対象） |
| バッファ | 缓冲 | 予備工数 |
| クリティカルパス | 关键路径 | 全体工期を決める最長経路 |
| 相見積もり | 多方报价对比 | 3社以上から取るのが一般的 |
| 稟議（りんぎ） | 审批流程 | 社内決裁プロセス |
| 定例（ていれい） | 定期例会 | お客様との定例会議 |

## 二、契約形態別の見積もり注意点

### 請負契約（請負）

**特徴**：
- 成果物の完成が義務
- 見積もり超過は受託者（開発側）の負担
- **必ずバッファを上乗せする**（30%以上推奨）

**見積もり時の注意**：
- 要件をできるだけ詳細に固めてから提示
- 「一式」などの曖昧表現は後で揉める原因になるため避ける
- 変更管理（Change Control）のプロセスも見積書に明記

### 準委任契約（準委任）

**特徴**：
- 「善管注意義務」で業務を遂行
- 成果物の完成ではなく「プロセス」が評価対象
- 工数ではなく「期間＋人数」で契約することが多い

**見積もり時の注意**：
- 月額単価 × 人数 × 期間 が基本
- スキルに応じて単価を変える（例：PM 150万円/月、SE 120万円/月、PG 90万円/月）
- 作業範囲（どこまでやるか/どこからは別料金か）を明示

## 三、日本の単価相場（参考値）

### 人月単価の目安（2024-2026年相場）

| 職種 | 単価帯（万円/人月） | 備考 |
|------|-------------------|------|
| プロジェクトマネージャー (PM) | 150〜250 | 経験・規模による |
| システムエンジニア (SE) | 100〜180 | 上流工程含む |
| プログラマー (PG) | 80〜130 | 製造・テスト中心 |
| AI/MLエンジニア | 130〜250 | 専門性高い |
| フルスタック（個人事業主） | 120〜200 | 一括請負の場合 |

### 諸経費

見積書に載せる諸経費の一般的な項目：

- **サーバ/インフラ費用**：AWS/GCPなど実費 ＋ 管理工数
- **外部API利用料**：kintoneライセンス、AI API（ChatGPT/Whisper等）
- **ドメイン・SSL**：年額数千〜数万円
- **交通費**：打ち合わせのある案件では別途計上
- **諸経費率**：直接費の10〜20% が一般的

### 保守運用費用の目安

- 開発総額の **15〜20%/年** が相場
- 内訳：監視、障害対応、軽微な改修、OS/ライブラリ更新
- 月額で見積もる場合：(開発費 × 0.18) ÷ 12

## 四、工程別工数配分の目安

大規模システム開発の標準的な比率（IPA SEC データより）：

| 工程 | 比率 | 備考 |
|------|------|------|
| 要件定義 | 10〜15% | ヒアリング、調査、分析 |
| 基本設計 | 15〜20% | アーキテクチャ、画面設計 |
| 詳細設計 | 15〜20% | DB設計、API設計 |
| 製造（コーディング） | 20〜25% | 実装 |
| 単体テスト | 10〜15% | |
| 結合テスト | 10〜15% | |
| 総合テスト | 5〜10% | |
| 導入・移行 | 5% | |

小型案件（1人全栈、〜30人日）ではこの比率は崩れる。
実装比率が高くなり（40〜50%）、要件定義や設計は簡略化される傾向がある。

## 五、日本特有の見積もり実務

### 見積書の標準構成

```
1. はじめに（提案の背景・目的）
2. 提案内容・システム概要
3. 見積もり条件
   - 前提条件（弊社作業範囲 と お客様作業範囲 を明示）
   - 作業範囲外の明示（将来フェーズ）
4. 費用内訳
   - 初期開発費用（工程別 × 工数 × 単価）
   - 月額費用（インフラ、API、保守）
   - 諸経費
   - 合計
5. スケジュール
6. 支払条件
7. 特記事項（変更管理、瑕疵担保、機密保持等）
```

### 失敗しやすいポイント（日本案件）

1. **間接工数を忘れる** — お客様定例、進捗報告会、稟議資料作成が想像以上に時間を食う
2. **「一式」と書く** — 後で「一式の中にXXが入ってない」と揉める
3. **バッファを公開しすぎる** — 「バッファ30%入ってます」と言うと「じゃあ削って」と言われる
4. **契約形態を確認しない** — 請負なのか準委任なのかでリスクが全然違う
5. **保守費用を忘れる** — 開発費だけ見積もって保守の話をしていないと後で揉める

### 日本客戶との見積もり交渉 Tips

- 安すぎる見積もりは「品質が低いのでは」と疑われる
- 相見積もり（3社比較）が当たり前なので、競合の存在を前提に
- 根拠（WBS、過去実績）を示せば値切られにくい
- 「なぜその工数なのか」を説明できることが最重要
- 月額費用は「定額制」を嫌がる日本企業も多いので、内訳明示
