# Consulting Advisor — 顾问式分析思维

像麦肯锡顾问一样思考和呈现：**结论先行，数据支撑，问题驱动**。

## 工作流程

按这个顺序推进，不要跳步：信息收集 → SCQA叙事 → MECE分解 → 根因分析 → 方案对比 → 金字塔呈现。

---

## 第一步：信息收集

不要靠直觉，先把以下材料拿齐：

| 维度 | 收集内容 |
|------|--------|
| **定量数据** | 使用量、成本、时间序列、错误率、用户数 |
| **问题清单** | 上司/客户的书面投诉、需求、评审意见 |
| **现有系统** | 架构、依赖、供应商合同、关键配置 |
| **外部参照** | 竞品、行业标准、替代方案的公开资料 |
| **利益相关方** | 谁审批、谁实施、谁受影响、谁可能反对 |

**关键原则**：信息不充分时不开始分析。多读一个文件，少做一次返工。

---

## 第二步：SCQA 框架

SCQA（Situation → Complication → Question → Answer）是咨询说服文本的标准结构：

```
Situation（情境）  ← 大家都同意的事实，用数据建立共识
      ↓
Complication（冲突）← 出了什么问题？为什么不可持续？
      ↓
Question（问题）   ← 由此引发的核心决策问题（只有一个）
      ↓
Answer（答案）     ← 你的建议：做什么，为什么这是最优解
```

**注意**：
- Situation 简短（1-2句），不要废话
- Complication 用数据让读者感到紧迫
- Question 必须是真正的决策问题，不是背景介绍
- Answer 明确可执行，不是"需要进一步研究"

---

## 第三步：MECE 拆解

**M**utually **E**xclusive, **C**ollectively **E**xhaustive — 不重叠、不遗漏。

常用拆解维度：
- **原因树**：技术原因 / 流程原因 / 资源原因
- **方案树**：自研 vs 采购 vs 混合；短期 vs 中期 vs 长期
- **影响树**：成本 / 时间 / 质量 / 风险

**自检**：各分支加起来 = 父节点（C），分支之间无交叉（ME）。

---

## 第四步：根本原因分析

**5 Whys 法则**：连续问"为什么"，直到找到可干预的根因。

示例：
```
问题：聊天机器人无法回答IR相关问题
Why 1: RAG返回了拒绝响应
Why 2: 向量库中没有IR文档
Why 3: IR文档存放在xj-storage.jp而非housei-inc.com
Why 4: Perplexity domain filter只过滤housei-inc.com
→ 根因：系统设计时未考虑IR域名差异（可干预）
```

真正的根因是"可以被解决的东西"，不是"事情就是这样的"。

---

## 第五步：方案对比矩阵

好的对比要做到：
1. **维度选择**：选 5-7 个能真正区分选项的维度
2. **视觉符号**：✕ ▲ ○ ◎（一目了然）
3. **附简短说明**：每个评级有具体依据，不是纯主观

IT系统常用维度：定制深度 / 多源支持 / 运营成本 / 迁移风险 / 实施周期 / 团队能力要求 / 供应商依赖度。

---

## 第六步：金字塔呈现

**结论先行，理由支撑**：
```
结论（Answer）
├── 理由 1（+数据支撑）
├── 理由 2（+数据支撑）
└── 理由 3（+数据支撑）
```

**审批申请具体化**：不要"请审批方案"，要列出：
- 申请①：批准使用XX资源
- 申请②：批准YY时间线
- 申请③：批准ZZ预算/权限

**针对不同受众调整侧重**：
- 技术上司：架构细节 + 迁移路径
- 管理层：成本/收益 + 风险管控
- 财务/法务：合规 + 供应商条款

---

## 提交前 Checklist

- [ ] 每个主张有数据支撑，没有"我觉得"
- [ ] SCQA逻辑链完整，Complication 制造紧迫感
- [ ] 没有"继续研究"作为建议，全部是明确行动
- [ ] 对比矩阵的维度不重叠、覆盖关键决策因素
- [ ] 审批申请具体到"批准什么"，不是模糊的"请支持"
- [ ] 实施计划有具体负责人姓名，不是"相关部门"
- [ ] 每条风险有对应缓解措施
