# 模式 B 操作手册：按已搭好的 harness 干活

> 这是 SKILL.md「模式 B：按 harness 作业」的完整展开。前提：仓库已经搭好 harness（分层 CLAUDE.md、
> 权限边界、可能有 hooks / skills / LSP）。你的任务不是再搭一套，而是**在这套现成的工作环境里高效地完成一件
> 具体的开发任务**——尤其是那种「一旦上手就容易改一半崩掉」的跨文件大改动。
>
> 核心心态来自源材料：**你不是在「开发」智能体，而是在用一个已经造好的车。** 你要做的是把路线规划好、
> 把油门和刹车用对，而不是边开边改车。

## 一、在子目录启动，不在仓库根

金句原文：**"Initializing in subdirectories, not at the repo root."**

大多数人习惯 `cd` 到仓库根再 `claude`。在大库里这等于一开机就把那份巨大的根 CLAUDE.md（所有微服务、所有
团队的规则）整个塞进 context——这些规则对你「只想改支付服务」这件事 90% 是噪音。

正确做法：**先 `cd` 到你真正要动的那个领域，再开 Claude。** 要改支付服务就：

```bash
cd services/payments
claude
```

这样做不会丢通用规则。Claude 仍然会**沿目录树往上走**，把一路上每一个 CLAUDE.md 都加载进来（根 CLAUDE.md
的跨包通用约定一个不少），但它会**优先 `services/payments/CLAUDE.md`**，整个 context 聚焦在这一个领域。

为什么这是低成本高回报：agentic search 「relies heavily on a good starting context」——Claude 靠 grep + 读文件
一步步摸清系统，起点越聚焦，它越不会在无关目录里乱翻烧 context。在子目录启动，就是直接把它放在了正确的起点上。

实操要点：

- **改哪个领域就进哪个领域的目录**，不要图省事一直待在根目录。跨多个领域的任务，见第四节的会话拆分。
- 进去之后顺手确认这个子目录的 `CLAUDE.md` 里有没有写清「这块怎么测、怎么 lint」。有就照它的命令跑——否则
  Claude 改完一个支付文件可能去跑**整个仓库的测试套件**（几十分钟、把 context 也烧光）。
- 这是全公司「最高 ROI 的三个动作」之一（另外两个是 CLAUDE.md 砍到 200 行以内、装 LSP）。零成本，立刻见效。

## 二、复杂任务先 Plan 后写

金句原文（Boris）：**"Pour your effort into the plan so Claude can one-shot the implementation."**

复杂任务最容易翻车的地方，不是「写得不够多」，而是「方向一开始就歪了，越写越偏」。所以把功夫花在 plan 上：
**先单独跑一轮，只敲定方案、不写一行实现代码；方案确认后，再切到 auto-accept，让 Claude 一发命中地把它实现出来。**

具体节奏：

1. **进 Plan Mode**（Boris 几乎所有复杂任务都从这里开始）。在这一轮里只做一件事：和 Claude 来回打磨方案——
   要改哪些文件、按什么顺序、有哪些边界情况、怎么验证。这一轮**不允许写实现代码**。
2. 方案你看着对了、心里有底了，再**切到 auto-accept**，让它按既定方案连续实现。因为方案已经在 context 里钉死，
   实现阶段它不需要再做大决策，「一发命中」的概率大幅提高。

为什么有效：方案阶段是「判断」，实现阶段是「执行」。把判断集中在一个干净、低消耗的回合里彻底想清楚，执行阶段就
变成机械落地——这正是把人的注意力用在刀刃上。**反过来，边想边写，等于让 Claude 在 context 已经被代码塞满、
注意力涣散的时候做最重要的决策，这是大改动改一半崩掉的根因之一。**

把这条和第四节连起来看：复杂任务里，「Plan」往往就是会话 1 的唯一产物，落成一个 plan 文件，给后面的实现会话当桥梁。

## 三、先派子 agent 探索，主 agent 留干净 context

在大库里，**「读懂这个系统是怎么工作的」本身就要烧掉几万 token**——要 `ls` 目录、grep 符号、来回读几十个文件，
每读一个才知道下一个该读哪个。如果让主 agent 自己干这件事，等它终于摸清楚，主 context 已经被几十个文件的原文塞满，
真正动手改代码时已经没多少干净空间了。

正解：**派一个子 agent 去探索，主 agent 的 context 保持干净。**

- 子 agent 在它**自己独立的 context window** 里读那几十个文件，把脏活全在隔离区干完；
- 它只**回传一份几百字的 findings**（结论、关键文件路径、约束、坑），主 agent 收到的是浓缩结论，不是几万 token 的原文。

最简单的下令方式，一句话就够：

> **「先用 subagent 调查我们项目里 X 是怎么实现的，写成 findings 文件，再回来动手改。」**

为什么要写成 **findings 文件**而不是只在对话里回传：文件能跨会话存活。探索的产物落盘后，第四节里后续的实现会话
可以从干净 context 重新加载它，不必重新探索一遍。findings 文件 + plan 文件，是会话拆分的两根桥梁。

实操要点：

- 探索任务越具体越好：「X 是怎么实现的」要落到具体的模块 / 数据流 / 调用链，别让子 agent 漫无目的地通读全库。
- 让子 agent 在 findings 里**带上关键文件的相对路径和行号区间**，主 agent 后续直接精确打开，不用再 grep 一遍。
- 探索和实现是两种不同的认知活动，混在一个 context 里会互相污染。子 agent 的本质就是给「探索」一个用完即弃的隔离上下文。

## 四、跨大量文件的改动 = 拆成多个会话，不是写更长的 prompt

这是模式 B 里最反直觉、也最关键的一条。

重构、迁移、跨服务的大改动有个共同的失败模式：**Claude 早期很稳，到后期开始忘事、漏改、出错。** 几乎所有人的
第一反应是「那我把 prompt 写得更长更细，把所有要求一次性交代清楚」——**这是错的方向。** 更长的 prompt 只会让
context 更快被填满，反而加速后期的崩塌。

正解：**把一个大改动拆成多个会话，每个会话从干净 context 开始，用文件（plan / findings）做桥梁串起来。**

一个典型拆法：

| 会话 | 干什么 | 产物 | 是否动代码 |
|---|---|---|---|
| 会话 1 | **只探索 + 写 plan**（配合第三节的子 agent） | 一个 plan 文件 | 不动代码 |
| 会话 2 | 加载 plan，**实现第一个模块**，把它的测试跑通 | 第一个模块的代码 + 绿测 | 动代码 |
| 会话 3 | 从干净 context 开始，加载 plan，**实现下一个模块** | 下一个模块的代码 + 绿测 | 动代码 |
| ……   | 每个模块一个新会话，循环 | | |

关键纪律：

- **每个会话都从干净 context 开始。** 不要在一个会话里硬扛完整个迁移——那等于回到「靠超长 context 撑全程」的老路。
- **plan 文件是桥梁。** 会话之间不靠「Claude 记得」来传递信息，靠落盘的 plan / findings 文件。会话 2 一开始就把
  plan 文件读进来，于是它知道全局蓝图，但 context 里只有它这一个模块需要的细节。
- **一个模块跑通测试再开下一个会话。** 每个会话有明确的、可验证的完成标志（绿测），而不是模糊地「改了一大片」。
  这样即使某个会话出了问题，回退面也只是一个模块。

把第二、三、四节连起来就是一条完整链路：**会话 1 用子 agent 探索 → 出 plan（第二、三节）；会话 2/3/… 各实现
一个模块（第四节）。** 这就是「Pour your effort into the plan so Claude can one-shot the implementation」在多会话
尺度上的落地。

## 五、大规模迁移用 /batch

当改动不是「几个模块串行做」而是「同一种改法要套到几十处」（典型如全库迁移、批量升级 API、统一替换某个模式），
人工一个会话一个会话推太慢。这时用 `/batch`——Boris 真实在用的多 agent 编排，一条命令搞定。

流程：

1. **先对话敲定迁移方案**（还是第二节的精神：先把 plan 谈清楚）。这一步决定「每一处都要怎么改、怎么自测、什么算改完」。
2. 方案定了，`/batch` **一次性派出几十个并行子 agent**。
3. 每个子 agent **在自己独立的 git worktree 里跑**——互不干扰、互不踩踏，各自改各自那一处。
4. 每个子 agent 改完**自己跑测试自测**，通过后**各自开一个 PR**。
5. 跑完，你拿到的是**一堆 PR，排队等 review**。

为什么独立 worktree 是关键：几十个 agent 同时改同一个工作区会互相覆盖、互相污染。每个跑在独立 worktree（独立分支、
独立工作目录）里，就能真正并行，而且每个改动天然是一个可独立 review、可独立回滚的 PR。这把「人审方案 + 机并行执行 +
人审结果」三段清晰切开：你只在头尾两端投入判断，中间的体力活交给并行 agent。

什么时候用 /batch、什么时候用第四节的会话拆分：

- **同构、可并行、彼此独立**的大批量改动 → `/batch`（几十处长得差不多、改完互不依赖）。
- **异构、有先后依赖、需要边做边判断**的大改动 → 第四节的会话拆分（模块之间有顺序、后一个依赖前一个的结果）。

## 六、权限用白名单，而非 --dangerously-skip-permissions

为了少被权限确认打断，很多人直接上 `--dangerously-skip-permissions`。**别这么干**——这等于把所有审计和护栏一次性
拆光，在真实仓库里一个误操作就可能是生产事故。

正解：**用 `/permissions` 预先把常用的安全命令放进白名单。**

- 把你高频且确定安全的命令（跑测试、lint、git status/diff、读文件这类）用 `/permissions` 一次性放行，之后它们
  不再反复弹确认；
- 而真正有风险的动作（删文件、改生产配置、推远端等）**仍然走确认 / 仍然在审计范围内**。

这就在「不被反复打断」和「保留权限审计」之间取得了平衡——你拿到了 skip-permissions 想要的顺畅，却没丢掉它代价
高昂的那部分安全性。团队层面，常用的放行规则和 `permissions.deny` 应该提交到 `.claude/settings.json` 里全队共享
（模板见 `assets/templates/settings.json`），新人一进来就自动有同一套权限边界。

## 七、创始人 Boris 的五条工作流

Boris 本人怎么用 Claude Code：**「我在终端里同时跑 5 个 Claude，外加 5~10 个 claude.ai/code，各干不同的任务。」**
这种高并行度不是靠蛮力，而是靠下面五条把每一路都收拾得干净可控：

1. **不用 `--dangerously-skip-permissions`。** 用 `/permissions` 白名单放行常用安全命令，既不被反复打断，又保留
   权限审计（即第六节）。
2. **几乎所有复杂任务都从 Plan Mode 开始。** 先把方案谈定，再切 auto-accept 一发命中地实现（即第二节）。
3. **挂一个 PostToolUse hook 自动格式化 Claude 写的代码。** Claude 大约有 10% 的格式没对齐，这个 hook 自动抹平，
   避免 CI 因为格式问题挂掉。
4. **一件事一天做超过一次，就把它做成 slash command / skill。** 他有一个 `/commit-push-pr`，一天用几十次。
   高频操作沉淀成命令 / 技能后，每次零认知负担、零 context 开销。
5. **团队共享一份持续打磨的 CLAUDE.md。** 放在 git 里全队共用，Claude 一犯错就立刻往里加一条；是一份活文件，越用越准。

贯穿这五条的**心态**才是重点：**不是「装上就用」，而是「每天打磨它」。** 把 harness 当成一个会进化的工作伙伴，
每天喂它新规则、新工具、新工作流——它会一天比一天更懂你的项目和你的团队。模式 B 里你不只是「使用」harness，
你也在用每天的小修小补让它变得更好用。

## 八、端到端例子：一个跨前后端的中等重构

**任务**：把用户资料接口从同步返回头像 URL，改成「后端返回头像对象 ID，前端按需异步拉取头像」。涉及后端
`services/profile/` 的接口与 DB 查询、前端 `web/src/features/profile/` 的数据获取与渲染。中等规模、跨前后端、
不是同构批量——所以走会话拆分，不走 `/batch`。

把上面七节串成一条完整流程：

**会话 1 · 在子目录启动 + 子 agent 探索 + 出 plan（不动代码）**

```bash
cd services/profile     # 第一节：进真正要动的领域，context 先聚焦后端
claude
```

进 Plan Mode，先派子 agent 摸清现状（第三节）：

> 先用 subagent 调查我们项目里用户资料的头像数据是怎么从后端流到前端渲染的：后端 `services/profile/` 里
> 头像 URL 在哪个接口、哪个 DB 查询产生，前端 `web/src/features/profile/` 里在哪里取、哪里渲染。
> 写成 findings 文件，带上关键文件的相对路径和行号，再回来。

拿到 findings 后，和 Claude 一起把方案谈死（第二节的「Pour your effort into the plan」），落成一个 plan 文件，
明确：要改哪些文件、后端先改还是前端先改、每一步怎么自测、什么算这一步完成。**这一轮到此为止，不写实现代码。**

**会话 2 · 实现后端模块，跑通后端测试**

从干净 context 重新开 Claude，仍在 `services/profile/`。先把 plan 文件和 findings 文件读进来（文件做桥梁，
第四节），然后只做后端这一块：改接口让它返回头像对象 ID、调整 DB 查询。按**这个子目录 CLAUDE.md 里写的后端测试
命令**自测（不是跑全库测试），绿了再停。

**会话 3 · 实现前端模块，跑通前端测试**

```bash
cd web/src/features/profile   # 换到前端领域，context 重新聚焦
claude
```

再次从干净 context 开始，读同一份 plan 文件，实现前端这一块：改成拿到对象 ID 后异步拉取头像、调整渲染与
loading 态。按前端子目录的测试 / lint 命令自测，绿了再停。

**贯穿全程的护栏**

- 权限：用 `/permissions` 预先放行跑测试、lint、git diff 这些（第六节），全程不用 skip-permissions。
- 格式：PostToolUse hook 自动格式化每一处改动（第七节第 3 条），不用操心 CI 因格式挂。
- 沉淀：如果这次发现「改完资料接口必须同时更新某个契约测试」这类**会重复出现**的规矩，就顺手加进对应子目录的
  CLAUDE.md（第七节第 5 条），让 harness 因为这次任务变得更准一点。

**这条流程的收益**：整个跨前后端重构从头到尾，**没有任何一个会话的 context 被几十个文件的原文 + 超长 prompt +
两端代码同时塞满**。探索的脏活在子 agent 隔离区干完、只回传几百字；全局蓝图钉在 plan 文件里、跨会话稳定传递；
每个会话只装它这一块需要的细节，并以「绿测」收口。这就是「把跨大量文件的改动拆成多个会话、而不是写更长的 prompt」
在一个真实任务上的样子。
