## 怎么用这份清单

两步，顺序不能反：

**第一步 · 先跑脚本拿机械体检结果。**

```bash
python ~/.claude/skills/harness-builder/scripts/audit_harness.py <repo_path>
```

它机械地查出：每个 CLAUDE.md 的行数（>200 标 FAIL、>150 标 WARN）、CLAUDE.md 里引用但已不存在的文件/路径（漂移）、
是否分层、`permissions.deny` 有无、地图有无、`.mcp.json` 是否在 skill/CLAUDE.md 还没打磨好时就出现、`.claude/HARNESS.md` 有无，
以及几个 Goodhart 高风险的正则信号（「覆盖率>85%」「行数<50」「圈复杂度」这类硬量化目标）。

**第二步 · 拿着脚本报告对照本清单做需要判断的部分。**

下面每一项都标了【机械可查 or 需判断】：

- 标【机械可查】的，**直接读脚本报告里对应的那一行，别自己再去翻文件数行数 / 试路径**——那是在烧 context 重做脚本已经做完的事。
  本清单对这些项只补充「脚本报错之后怎么修」。
- 标【需判断】的，是脚本查不了、必须由你读内容来判断的（规则是否过期、是否过度显形、档位是否错配、高频操作是否该沉淀）。**你的 context 优先花在这些上。**

每项的结构统一是：【检查什么】【怎么判定合格/不合格】【不合格怎么修】【机械可查 or 需判断】。

## 分组逐项清单

### 1. CLAUDE.md

**1.1 单文件 ≤200 行**
- 检查什么：每个 CLAUDE.md 的行数。
- 判定：脚本里 >200 行报 FAIL，>150 行报 WARN（逼近上限）。≤150 合格。理由：CLAUDE.md 每次会话全量进 context，越长越占工作空间，过了 ~200 行 Claude 明显开始忽略指令。
- 不合格怎么修：用「删除测试」逐行删（见 1.3），删不动就分层（见 1.2），把模块细节下沉到子目录 CLAUDE.md 或 `references/`。瘦身写法详见 references/claude-md-guide.md。
- 机械可查 or 需判断：**机械可查**（读脚本「CLAUDE.md 过长/偏长」那几行）。

**1.2 是否分层**
- 检查什么：是否只有根 CLAUDE.md 在硬扛全部细节，还是根放指针、子目录放模块细节。
- 判定：大库里只有一个根 CLAUDE.md（脚本报 WARN「未分层」）= 不合格。合格形态是「根目录的 CLAUDE.md 只放跨包通用约定 + 关键的坑 + 指针，其他细节都会变成噪音」，每个值得的子目录有自己的 CLAUDE.md。
- 不合格怎么修：把根里的模块级细节切到对应子目录的 CLAUDE.md（模板 assets/templates/CLAUDE.subdir.md）；根目录只留指针。Claude 会沿目录树往上逐级加载，通用规则不会丢。
- 机械可查 or 需判断：**有无机械可查**（脚本报「未分层/已分层」）；**「根目录里这条到底算指针还是该下沉的细节」需判断**——逐条问：这行是不是只对某个子目录有用？是就下沉。

**1.3 有没有「删除测试」该删的行**
- 检查什么：每一行规则是否还配得上它占的 context。
- 判定：对每行做删除测试——「如果我删掉这行，Claude 还会做对吗？」答案是「会」（属于常识 / 代码里已经体现 / 工具链已经强制）→ 这行是噪音，该删；只有答案是「不会」的行才留。Boris 原话：**Ruthlessly edit your CLAUDE.md over time.**
- 不合格怎么修：删。注意一个反模式：Claude 反复犯同一个错时，先别急着加规则——**先看 CLAUDE.md 是不是已经太长、把真正的规则淹了**，往往是删几行而不是加一行。
- 机械可查 or 需判断：**需判断**（脚本数不出「这行有没有信息量」，只能由你逐行读着判断）。

**1.4 引用路径是否漂移**
- 检查什么：CLAUDE.md 里 ``` `path/to/file` ``` 或 `[..](path)` 形式引用的仓库内文件，是否还存在。
- 判定：引用了已删除 / 已改名的文件 = 漂移 = 不合格（脚本报 FAIL「CLAUDE.md 引用漂移」并列出具体路径）。这类引用会把 Claude 导向不存在的地方，浪费 context 找空气。
- 不合格怎么修：按脚本列出的路径，逐个改成新路径或删掉该引用。
- 机械可查 or 需判断：**机械可查**（直接照脚本列出的漂移路径修，别自己再去 grep 验证存在性）。

**1.5 是否过期（为旧模型写的枷锁）**
- 检查什么：有没有当年为绕开旧模型弱点而写、如今已成枷锁的规则。典型：「一次重构只改一个文件」「禁止跨文件编辑」这类——现在的模型早就能稳定做跨文件改动，这条反而在拖后腿。
- 判定：一条规则若是「为了规避某个模型当年的能力短板」而非「为了这个仓库的真实约束」，且那个短板已不存在 = 过期 = 不合格。
- 不合格怎么修：删除或放宽。删之前确认它针对的是模型能力（可删）而不是仓库的真实业务约束（要留）。
- 机械可查 or 需判断：**需判断**（要懂「这条当年是为了什么」+「现在模型还吃这套吗」，脚本查不了）。

### 2. 权限与忽略

**2.1 `permissions.deny` 是否覆盖生产库 / 密钥**
- 检查什么：`.claude/settings.json` 的 `permissions.deny` 有没有挡住生产数据库、密钥/凭据目录、`.env`、部署/发布命令等危险面。
- 判定：`permissions.deny` 为空（脚本报 WARN「权限边界薄弱」）或没覆盖到生产库/密钥 = 不合格。合格：危险操作进黑名单，而不是靠 `--dangerously-skip-permissions`。
- 不合格怎么修：用 assets/templates/settings.json 起一份，按本仓库实际把生产 DB、密钥路径、危险命令加进 `deny`；常用安全命令用 `/permissions` 或 allow 放行，保留审计。
- 机械可查 or 需判断：**有无机械可查**（脚本报 deny 条数）；**「这几条 deny 够不够覆盖本仓库的危险面」需判断**——脚本不知道你的生产库叫什么。

**2.2 生成物 · 构建产物 · 第三方是否排除**
- 检查什么：`node_modules`、`dist`/`build`/`target`/`out`、生成的代码、`vendor` 第三方等，是否被忽略，不让 Claude 把 context 烧在这些目录上。
- 判定：这些目录没被排除 = 不合格。它们会让 agentic search 在垃圾里翻找、地图和 grep 全是噪音。
- 不合格怎么修：加 `.ignore` 风格的排除规则 / 在 `permissions.deny` 里挡掉这些路径。
- 机械可查 or 需判断：**需判断**（脚本本身的遍历会跳过这些目录，但不报告本仓库是否在配置层面排除了它们；用 recon.py 的构建产物/生成目录清单对照着补）。

**2.3 规则是否提交进 `.claude/settings.json` 供团队共享**
- 检查什么：忽略/权限规则是放在能进 git 的 `.claude/settings.json`（团队共享），还是只躺在某人本地的 `settings.local.json`。
- 判定：规则只在本地、没进版本库 = 不合格（脚本若连 `.claude/settings.json` 都找不到会报 WARN「无 settings.json」）。harness 的价值之一就是让全队自动拿到同一套排除。
- 不合格怎么修：把团队该共享的 `permissions.deny` / 忽略规则放进 `.claude/settings.json` 并提交；纯个人偏好才留在 `settings.local.json`。
- 机械可查 or 需判断：**文件有无机械可查**；**「哪些规则该升到共享层」需判断**。

### 3. 代码库地图

**3.1 地图是否与真实顶层目录一致**
- 检查什么：根目录那张代码库地图（列出每个顶层目录一句话说明）是否还对得上真实的顶层目录——目录增删/改名后地图有没有同步更新。
- 判定：地图列了已删的目录、漏了新增的目录、或描述与现状不符 = 不合格（地图过期比没有地图更坑，会把 Claude 直接导歪）。脚本只能查地图「有没有」，查不了「准不准」。
- 不合格怎么修：用 recon.py 输出的真实顶层目录清单，对照地图逐条核：删掉已不存在的、补上新增的、修正过时描述。模板见 assets/templates/codebase-map.md。
- 机械可查 or 需判断：**有无机械可查**（脚本报地图有无）；**「内容是否与真实目录一致」需判断**（拿 recon.py 的目录清单对一遍）。

### 4. Hooks

**4.1 是否只停留在「防错」，有没有用上「自进化」**
- 检查什么：现有 hooks 是不是全是 PostToolUse 自动格式化 / lint 这类「防 Claude 做错」的，有没有用上让整套设置自我进化的那一类。
- 判定：只有防错 hook、没有自进化 hook = 没用满（不算硬不合格，但是高价值缺口）。Hooks 真正的价值是**让整套设置自我进化**，不只是挡错：
  - **Stop 反思回写**：会话结束时自动反思「Claude 这次有没有重复犯某个错？要不要写进 CLAUDE.md？」并由 hook 自己改 CLAUDE.md。
  - **SessionStart 注入**：按当前子目录动态注入模块 context（进 `payments/` 注入支付 skill，进 `auth/` 注入 auth context）。
- 不合格怎么修：从 assets/templates/hooks-examples.md 里挑 Stop 反思回写、SessionStart 注入挂上。**先挂能自进化的那一两个，别一次堆满。**
- 机械可查 or 需判断：**需判断**（要读 hook 在干什么、判断属于防错还是自进化）。

**4.2 自动回写 CLAUDE.md 的产物是否被纳入 200 行纪律**
- 检查什么：如果挂了 Stop 反思回写 hook，它往 CLAUDE.md 里追加的内容有没有让文件慢慢超 200 行。
- 判定：自进化 hook 只加不删、把 CLAUDE.md 越喂越长 = 不合格。自进化不能突破瘦身纪律，否则越「进化」越垃圾。
- 不合格怎么修：把这些自动追加的行也拉进「删除测试」一起删；或让回写 hook 带去重 / 触顶提醒；超 200 行就触发分层。
- 机械可查 or 需判断：**行数机械可查**（脚本会报这个 CLAUDE.md 是否超 200）；**「哪些追加行该删」需判断**。

### 5. Skills

**5.1 高频操作是否已沉淀为 skill**
- 检查什么：团队/个人「一天会做不止一次」的操作（DB 迁移、微服务标准发布流程、commit-push-pr 等）是否已经沉淀成 skill，而不是每次重新口述。
- 判定：存在「一天做不止一次、却还没成 skill」的操作 = 缺口。判据就是 Boris 那句：**If you do something more than once a day, make it a skill.** skill 按需加载（progressive disclosure），不像 CLAUDE.md 每次全量进 context，所以沉淀进 skill 几乎零 context 成本。
- 不合格怎么修：把该高频操作写成 skill（SKILL.md + 必要的脚本/模板）。怎么判断、怎么写见 references/harness-layers.md 的 Skills 节。
- 机械可查 or 需判断：**需判断**（脚本不知道你一天重复做了什么；这要靠回忆近期会话 / 看历史命令）。

**5.2 skill 是否该绑定子目录路径以免 context 污染**
- 检查什么：某些 skill 只对特定子目录有意义（支付部署 skill 只在 `services/payments/`），是否被绑定到了对应路径。
- 判定：只在某子目录有意义的 skill 却全局可见、在无关任务里也被纳入候选 = context 污染 = 不合格。
- 不合格怎么修：把这类 skill 绑定到对应子目录路径，让它只在相关目录的任务里出现。
- 机械可查 or 需判断：**需判断**（要懂这个 skill 的语义边界）。

### 6. 搭建顺序（MCP 是否上早了）

**6.1 MCP 是否在 CLAUDE.md / skill 还没打磨好时就接了**
- 检查什么：是否接了 MCP（Slack/Jira/DB/监控等）但 CLAUDE.md 或 skill 还没打磨好。
- 判定：有 `.mcp.json` 但 CLAUDE.md/skill 不到位（脚本报 WARN「MCP 可能上早了」）= 不合格。MCP 是**最后一层**；基础没搭好就接 MCP，接进来的全是噪音。正确顺序：打磨 CLAUDE.md + skills → plugin 打包 → 最后才上 MCP。
- 不合格怎么修：先把 CLAUDE.md（≤200 行、分层）和高频 skill 打磨到位，再保留/扩展 MCP；必要时先把 MCP 摘掉，等基础稳了再接回。
- 机械可查 or 需判断：**有无机械可查**（脚本查 `.mcp.json` 与 skill/CLAUDE.md 的先后）；**「接进来的 MCP 数据到底是不是噪音」需判断**。

### 7. 子目录测试 / lint 命令

**7.1 子目录测试 / lint 命令是否齐**
- 检查什么：每个子目录的 CLAUDE.md 有没有写清「这块用什么命令测、怎么 lint」，做到改哪块只跑哪块。
- 判定：子目录没写专属测试/lint 命令 = 不合格。否则 Claude 改一个支付文件就去跑**全量**测试套件（几十分钟、把 context 烧光），或者干脆不验证。
- 不合格怎么修：在每个子目录 CLAUDE.md 补上「只测/只 lint 这一块」的精确命令（模板 assets/templates/CLAUDE.subdir.md）。
- 机械可查 or 需判断：**需判断**（脚本不懂你的测试命令对不对、覆盖不覆盖这个子目录；要读子目录 CLAUDE.md 判断）。

### 8. 判断层

判断层最容易被忽略、也最危险，体检时务必对照 references/judgment-and-taste.md。

**8.1 有没有 Goodhart 化的量化硬目标**
- 检查什么：CLAUDE.md / 规则里有没有把覆盖率、行数、圈复杂度这类**纯量化指标**写成必须达成的硬目标。
- 判定：出现「覆盖率>85%」「单文件<50 行」「圈复杂度<N」当目标 = 不合格。**When a measure becomes a target, it ceases to be a good measure**——指标一旦成目标就会被钻空子：覆盖率会被 `assert True` 刷满、行数会被硬拆函数（行数合规但可读性塌）。不是 AI 坏，是优化器的本性。
- 不合格怎么修：按**判断力光谱 S1–S5** 重新归档——量化指标属 S2（Goodhart 高风险），只做参考、别当目标；真正想要的「质量」用 S3 验收标准即代码（语义条款）或 S4 偏好陈述（Project Taste）表达。
- 机械可查 or 需判断：**信号机械可查**（脚本有覆盖率/行数/圈复杂度的正则，报 WARN「可能的 Goodhart 化规则」）；**「这条是否真被当成硬目标、该挪到哪一档」需判断**（正则会误报，最终归档要人定）。

**8.2 有没有过度显形（把该留白的写死）**
- 检查什么：有没有把**战略方向感、价值观底线、带疤痕的禁忌、审美的争论**这四样「刻意不刻」的东西也写成了显式规则。
- 判定：把它们写死 = 过度显形 = 不合格。原因（不可能三角：Spec 完备性、Goodhart 抗性、Tacit 保全三者最多得其二）：
  - 🧭 战略方向感——外包了方向，你就失去了存在的理由。
  - 🚫 价值观底线——规则会制造「没禁止就是允许」；价值观恰恰在规则沉默处说话。
  - 🩹 带疤痕的禁忌——「别在周四下午发版」（三年前的事故）；把肌肉记忆般的禁忌硬讲出道理，它就死了。
  - 🎨 审美的争论——两个资深为命名吵一下午，这本身是组织健康的信号，不该被一条规则终结。
- 不合格怎么修：把这些条目从硬规则里撤掉，留白。Borges 1:1 地图 / 黑森林寓言：完美的显形等于没有显形；别把「好代码」窄化成「通过一组 Eval」。
- 机械可查 or 需判断：**需判断**（要识别一条规则承载的是战略/价值观/禁忌/审美，脚本完全查不了）。

**8.3 规则的 S1–S5 档位有没有错配**
- 检查什么：每条规则落的档位对不对。参照表：

  | 档 | 形态 | 示例 | Goodhart 风险 |
  |---|---|---|---|
  | S1 硬规则 | 布尔 | Lint、编译错误、类型系统 | 极低 |
  | S2 量化指标 | 连续值 | 覆盖率、圈复杂度 | 极高 |
  | S3 结构化规格 | 语义条款 | Spec、验收标准、行为契约 | 中等 |
  | S4 偏好陈述 | 软约束 | 「我们偏好 X 胜过 Y」 | 低 |
  | S5 裸直觉 | 不可文本化 | 「就是觉得不对」 | 免疫 |

- 判定：错配 = 不合格。三种典型错配：**把合规（本该 S4 偏好）放进 S1 硬规则**是失职；**把审美写成 S1**是暴力；**把战略拿去 S2 当 KPI**是悲剧。
- 不合格怎么修：逐条改挂正确档位——硬规则（Lint/类型）留 S1；语义验收挪到 S3；团队偏好挪到 S4 / Project Taste（assets/templates/project-taste.md）；裸直觉（S5）就别硬写。
- 机械可查 or 需判断：**需判断**。

### 9. 复审周期

**9.1 距上次完整复审是否超过 3–6 个月**
- 检查什么：上一次对整套 harness（尤其 CLAUDE.md）做完整复审是什么时候。
- 判定：超过 3–6 个月没完整复审 = 不合格。模型在进化，旧约束可能变枷锁、为补旧模型弱点写的 hook/skill 可能成了死重量，需要定期回看。
- 不合格怎么修：现在就做一轮完整复审（即本清单全过一遍），并把下次复审日期记进 HARNESS.md。
- 机械可查 or 需判断：**需判断**（脚本不存复审日期；从 HARNESS.md 的复审记录 / git 历史推断）。

**9.2 有没有 HARNESS.md 记录**
- 检查什么：根目录 `.claude/HARNESS.md` 是否存在，是否记录了搭了哪些层、各层文件位置、为什么这么搭、复审周期与上次复审日期。
- 判定：没有 HARNESS.md（脚本报 INFO「无」）= 不合格——它是模式 C 纠偏的入口，没有它每次体检都要从头摸。
- 不合格怎么修：补一份 HARNESS.md，把本次体检结论和下次复审日期写进去。
- 机械可查 or 需判断：**有无机械可查**（脚本报 HARNESS.md 有无）；**「内容是否完整、记录是否最新」需判断**。

## 纠偏报告输出模板

把脚本的 FAIL/WARN 项 + 上面需判断项的结论，汇总成一份**分优先级**的偏差报告。每条按「现状 → 问题 → 建议改动 → 为什么」四段写，让用户看得懂、能审批：

- **高（先修）**：会直接坑到 Claude / 有安全风险的。例：CLAUDE.md 超 200 行、引用漂移、`permissions.deny` 没挡住生产库或密钥、地图与真实目录对不上。
- **中（应修）**：拖累效率 / 用没用满的。例：未分层、子目录缺测试命令、高频操作没沉淀成 skill、只有防错 hook 没自进化、MCP 上早了。
- **低（可改进）**：判断层打磨与长期健康。例：S1–S5 档位微调、Project Taste 起草/补充、HARNESS.md 完善、复审周期补登记。

每条格式：

```
### [高 / 中 / 低] 一句话标题
- 现状：脚本/检查里看到的事实（引用具体文件与行数/路径，别泛泛而谈）
- 问题：为什么这是个问题（它在烧 context / 导向错误 / 会被 Goodhart 化 / 过度显形……）
- 建议改动：具体改哪个文件、删/加/挪什么（引用 assets/templates/ 里的模板）
- 为什么：这么改解决了上面的问题、对得上哪条 harness 原则
```

**强调：报告只是方案，改动要走审批门，不要直接大改。**
本技能默认「Plan → 审批 → 落地」：把分优先级报告交给用户确认，再按模式 A 步骤 3 的逐层审批门一项项落地——尤其在一个已有的真实仓库里，直接大改是危险的。落地后更新 HARNESS.md 的复审记录。
