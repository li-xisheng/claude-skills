## 1. 为什么 200 行是条线

CLAUDE.md 和 skill / reference 最大的区别是：**它每次会话都被全量加载进 context，无论这次任务用不用得上。** skill 是按需加载（progressive disclosure），CLAUDE.md 没有这个待遇。

由此推出两条因果：

- **写太长 = 跟自己抢 context。** 你在 CLAUDE.md 里多塞 100 行用不上的细节，就等于每次开会话先把工作区占掉一块。Claude 真正要用来读代码、推理、改文件的空间就少了——这正是「context 老爆不是模型小，是 harness 没搭好」的一个直接来源。
- **超过 ~200 行后，Claude 忽略指令的概率肉眼可见地上升。** 指令被淹没在长文里，重要的坑和不重要的碎碎念占同样的权重，模型很难分辨哪条是「违反会出事」。文件越长，越像噪音，关键约定反而失效。

所以 200 行不是一个美观偏好，而是一条功能性的线：**过线之后，你写的规则开始失效，而且还在持续偷你的 context。** 把单个 CLAUDE.md 控制在 200 行以内，是让规则真正起作用的前提。

> 这条线对每一个 CLAUDE.md 都成立——根目录的、每个子目录的，各自独立 ≤200 行。不是「全仓加起来 200 行」。

## 2. 分层加载：根目录只放指针和关键的坑

Claude 在某个目录启动时，会**自动从当前目录往上走目录树，把沿途每一个 CLAUDE.md 都加载进来**。这是分层加载的机制基础，也是为什么应该在子目录启动而不是仓库根。

利用这个机制，把 CLAUDE.md 拆成两类角色：

- **根 CLAUDE.md = 跨包通用约定 + 关键的坑 + 指针。** 它是「全仓都适用的那一点点」加「一张通往细节的地图」。一句话原则：**「根目录的 CLAUDE.md 应该只放指针和关键的坑，其他细节都会变成噪音。」**
- **子目录 CLAUDE.md = 这个模块的细节。** 测试命令、模块特有的坑、入口与数据流——只在干这块活时才需要，就让它只在进到这块时才被加载。

举例：你 `cd services/payments` 再开 Claude，它会加载 `services/payments/CLAUDE.md`（支付模块细节），同时往上走加载根 `CLAUDE.md`（全仓通用规则）。**通用规则一条没丢，但 context 聚焦在支付这一个领域**，而不是被所有微服务的规则一起塞满。

这就是为什么细节必须下沉：如果你把支付、风控、订单的细节全堆进根 CLAUDE.md，那么任何人改任何一个模块，都得先吞下另外两个模块的细节当噪音。分层加载就是为了不让这件事发生。

## 3. 删除测试：每一行都要挣得它的位置

源材料里 Boris 的原话：**「Ruthlessly edit your CLAUDE.md over time.」** 落到操作上，就是对 CLAUDE.md 里**每一行**问一个问题：

> **「删掉这行，Claude 还会按这条做对吗？」**
> - 会 → **删掉。** 这条是常识、或者代码本身已经体现了，写出来只是占位置、稀释信号。
> - 不会 → **才留下。** 只有「不写它 Claude 就会做错」的行，才值得占用每次会话的全量 context。

判断的关键是分清「Claude 已经知道的」和「这个项目特有、Claude 无从知道的」。

**该删的正例**（删掉 Claude 照样做对）：

- `写代码要有可读性` —— 常识，删。
- `函数应该单一职责` —— 常识，删。
- `使用 TypeScript 严格模式` —— 如果 `tsconfig.json` 里已经 `"strict": true`，代码已经体现了，删。
- `记得处理错误` —— 太泛，等于没说；要留也得是项目特有的错误处理约定（见下）。
- `提交前运行测试` —— 如果有 pre-commit hook 或 CI 卡着，机制已经保证了，删。

**该留的反例**（删了 Claude 大概率做错，因为它无从知道）：

- `生产数据库严禁直接操作，只能改 migration 文件由 CI 执行` —— 项目特有的红线，违反会出事。
- `这里的金额单位是「分」不是「元」` —— Claude 无从猜测，猜错就是资损 bug。
- `所有 API 响应必须走统一信封 { success, data, error }` —— 团队约定，不写它就会冒出五花八门的返回格式。
- `禁止出现 handler2 / utilNew / commonV3 这类命名` —— 这是这个团队的口味，模型默认不会这么严。
- `改完只跑 src/foo 的测试，别跑全量（全量要 20 分钟）` —— 项目特有、能省大量 context 的操作知识。

一句判据：**写常识是在稀释信号，写项目特有的坑是在加信号。** 删除测试就是反复用这把尺子刮掉稀释项。

## 4. 根 CLAUDE.md 该放 / 不该放

| 该放（信号） | 不该放（噪音 / 该下沉） |
|---|---|
| 全仓通用、跨所有包都成立的约定（不可变优先、语义命名、错误显式处理） | 某一个模块才有的约定（下沉到该模块的子目录 CLAUDE.md） |
| 关键的坑：违反会出事的红线（别碰生产库、别读 `.env` 密钥、发版前必须 lint） | 编程常识（代码要可读、函数要单一职责）——删除测试一律删掉 |
| 指针：去哪干活、细节在哪个文件（`改前端 → cd src，见 src/CLAUDE.md`） | 某个具体模块的测试/lint 命令（属于子目录） |
| 全局命令的入口（全量 lint / 全量测试怎么跑，并提醒优先用子目录更快的命令） | 长篇的架构介绍、历史沿革、设计动机（放 docs 或 reference，不占每次会话的 context） |
| 通往其他 harness 件的指针（`team 口味见 .claude/project-taste.md`、`目录速览见 CODEBASE_MAP.md`） | 已被 `tsconfig` / lint 配置 / CI 等机制保证的规则（机制已经体现，写了是冗余） |

记住根文件的两个职能：**「全仓通用的那一点点」+「通往细节的地图」**。它越像地图、越不像百科全书，就越好用。

## 5. 子目录 CLAUDE.md 该放什么

子目录 CLAUDE.md 只写**增量**——这个模块特有、根目录没有的东西。因为 Claude 会自动把根 CLAUDE.md 也加载进来，所以在子目录里重复通用约定，又是在制造噪音。

按重要性排序，子目录 CLAUDE.md 优先放这四样：

1. **本模块的测试 / lint / 类型检查命令（最重要）。** 写清「只测这个模块」的命令，例如 `npm run test -- src/foo`。
   不写这条的代价很具体：**Claude 改了一个支付文件，却跑起整仓测试套件，几十分钟、把 context 烧光。** 每个子目录都标明「这块怎么测、怎么 lint」，就是在止血。
2. **模块特有的约定与坑。** 例：`金额单位是「分」`、`所有响应走统一信封`、`这个模块不允许直接访问 DB，必须走 repository`。这些是删除测试里典型的「该留」项。
3. **关键入口与数据流。** 给 agentic search 一个好起点，省得它从头 `ls` + grep 摸索：`请求入口 server/index.ts → 路由 server/routes/* → service → repository`。一两行就够，别画长篇架构图。
4. **可沉淀为 skill 的高频操作。** 如果某个操作「一天会做不止一次」（如「新增一个 API 端点的标准步骤」），先在这里记下步骤，再考虑把它抽成一个绑定到本路径的 skill——把高频 SOP 从「每次全量加载的 CLAUDE.md」挪到「按需加载的 skill」，是省 context 的标准动作。

## 6. 反复犯错？先查长度，别急着加规则

最反直觉、也最容易踩的一条：**发现 Claude 反复犯某个错，第一反应不该是「加一条规则」。先去看 CLAUDE.md 是不是已经太长，把规则淹没了。**

因果链是这样的：文件越长 → 单条指令权重越低 → Claude 越容易忽略 → 你越想加规则去强调 → 文件更长 → 更容易被忽略。**加规则会让问题加速恶化，而不是解决。** 这是一个正反馈陷阱。

正确的处理顺序：

1. **先量长度。** 这个 CLAUDE.md 是不是已经接近或超过 200 行？（可以直接用 `scripts/audit_harness.py` 机械查行数。）
2. **如果偏长，先做删除测试瘦身。** 把稀释信号的常识行删掉，让真正的规则浮上来。常常瘦身之后，那条「一直被忽略的规则」自己就生效了——因为它终于不再被噪音淹没。
3. **瘦身后仍然犯错，再考虑加规则。** 而且要想：这条规则更适合放在哪一档？是不是其实该用一个 hook 来机械保证（比如格式问题用 PostToolUse 自动格式化），而不是靠一条文字规则去提醒？

**先减后加**，是维护 CLAUDE.md 的默认姿势。

## 7. 3–6 个月完整复审：模型在进化，旧规则会变枷锁

CLAUDE.md 不是「写一次放着」的文件。**每 3–6 个月，完整地从头到尾过一遍。** 理由是模型在持续进化，而你的规则是为「写下它那天的模型」量身定的——模型升级后，有些规则会从「拐杖」变成「枷锁」：

- **为旧模型弱点写的约束，现在可能反而拖后腿。** 经典例子：`每次重构只改一个文件`。这条是当年为了防止弱模型在跨文件改动中改崩而设的护栏。但今天的模型已经能稳稳地做跨文件编辑，这条规则反而把它绑死在低效的工作方式上——本该一次改完的重构被你强行拆成单文件。
- **为旧模型弱点写的 hook / skill，升级后可能成了多余负担。** 比如某个专门纠正旧模型某类常犯错误的 PostToolUse 检查，新模型早就不犯了，这个 hook 就只剩开销，没有收益，甚至偶尔误伤。

复审时对每条规则、每个 hook、每个 skill 多问一句：**「这是给哪个时代的模型写的？它现在还在解决一个真实存在的问题吗？」** 不再解决真问题的，删掉或放宽。复审同时也是一次大规模的删除测试。

> 把复审周期写进 `.claude/HARNESS.md`，并在每个 CLAUDE.md 底部留一行复审提示（模板里已经有：`<!-- 复审：每 3–6 个月完整过一遍 -->`），免得忘记。

## 8. 把它当活文件：Anthropic 团队的实践

Anthropic 内部 Claude Code 团队的做法值得直接抄：

- **整个团队共享同一份 CLAUDE.md，提交进 git。** 它不是某个人本地的私货，而是团队共同维护的资产，所有人都受益、所有人都能改。
- **一发现 Claude 做错了，就立刻把对应的规则加进去。** 不攒、不等下次专门整理——当场补上，让同样的错不再发生第二次。
- **它是一份持续打磨的活文件。** 心态不是「装好就用」，而是「每天打磨」：把 CLAUDE.md 当成一个在不断学习的队友，今天教它一条新规矩，明天删掉一条过期的。

这就是为什么前面所有规则要配合使用：**当场加新规则（活文件）** 必须配上 **删除测试 + 定期复审 + 反复犯错先查长度**，否则活文件会单调地变长、变臃肿、最终失效。**「随时加」和「狠心删」是一对，缺一不可。**

## 9. 模板怎么用

两份模板直接套用，把 `<尖括号>` 占位符换成项目真实内容，用不到的小节整段删掉（删比留更符合本指南精神）。

- **`assets/templates/CLAUDE.root.md`** —— 根目录用。模板头部已经写明三条铁律（≤200 行 / 只放指针和关键的坑 / 每行过删除测试），正文给了五个小节骨架：
  - `这是什么`（一两句项目定位 + 技术栈一句话）
  - `关键的坑`（违反会出事，最高优先级）
  - `跨包通用约定`（不可变、语义命名、错误显式处理这类全仓适用的）
  - `在哪干活`（**纯指针**：改前端 `cd src`、改后端 `cd server`、地图在哪、口味文件在哪）
  - `怎么验证`（只放全局命令，并提醒优先用子目录的更快命令）

  填的时候时刻提醒自己：每个小节都可能被删除测试刮掉几行，宁可少填。

- **`assets/templates/CLAUDE.subdir.md`** —— 每个值得的子目录用（不是每个目录都需要，只有有模块特有信息的才建)。模板头部强调「只写增量，通用约定留根目录别重复」。四个小节正好对应第 5 节的四样东西：
  - `这个模块负责什么`（一两句职责边界）
  - `怎么测 / 怎么 lint`（**最重要那条**，写清只测本模块的命令）
  - `本模块特有的约定 / 坑`
  - `关键入口与数据流`（可选）+ `本模块的高频操作`（可考虑沉淀为 skill）

填完后，对每份文件做一次删除测试 + 数一下行数，确认都 ≤200 行再交。

## 10. 臃肿反例 vs 精炼正例

同一个项目的根 CLAUDE.md，看看差别。

### ❌ 臃肿的反例（信号被噪音淹没）

```markdown
# MyApp

本项目是一个现代化的 Web 应用，采用了业界领先的技术栈。我们非常注重代码质量，
所有开发者都应该写出高质量、可读性强、可维护的代码。我们相信好的代码是团队的资产。

## 编码规范
- 请始终编写清晰可读的代码。
- 函数应该保持单一职责，尽量简短。
- 变量命名要有意义，不要用 a、b、c 这种。
- 请记得给复杂逻辑加注释。
- 使用 TypeScript，并开启严格模式（我们的 tsconfig 已经配了 strict）。
- 缩进使用 2 个空格（.editorconfig 已经配了）。
- 请处理好错误，不要让程序崩溃。
- 提交代码前请运行测试和 lint（CI 也会跑）。

## 架构介绍
本项目最初在 2021 年由三人小组发起，最早用的是 React 16，后来迁移到了 React 18，
当时为了支持并发特性我们做了大量改造……（此处省略 60 行历史沿革与设计动机）

## 支付模块
支付模块在 services/payments 下，金额单位是分。新增端点要先写 service 再写 route……
（又 40 行支付细节）

## 风控模块
……（再 40 行）
```

问题诊断：

- 开头一整段是空洞的价值宣言，删除测试全删——不写 Claude 照样会努力写好代码。
- `编码规范` 里清一色常识，且 `strict` / 缩进 / `提交前 lint` 都已被配置和 CI 机制保证，全是冗余。
- `架构介绍` 60 行历史沿革对当下任务零帮助，纯占 context，应进 docs。
- 支付、风控细节被堆在根目录 → 任何人改任何模块都得吞下另外两个模块的细节。**这正是细节没下沉的典型病灶。**
- 结果：文件三四百行，真正的红线（如果有的话）被埋在中间，Claude 大概率忽略。

### ✅ 精炼的正例（每行都挣得了位置）

```markdown
# MyApp — 给 Claude 的项目说明

## 这是什么
现代化 Web 应用，前端 React18+TS，后端 Node/Express。

## 关键的坑（违反会出事）
- 生产数据库严禁直接操作；只能改 migration 文件，由 CI 执行。
- `server/.env` 内密钥严禁读取、打印、提交。

## 跨包通用约定
- 不可变优先：返回新对象而非原地修改。
- 语义命名：禁止 handler2 / utilNew / commonV3。
- 错误必须显式处理，禁止静默吞掉。

## 在哪干活（指针）
- 改支付 → `cd services/payments`，细节见该目录 CLAUDE.md
- 改风控 → `cd services/risk`，细节见该目录 CLAUDE.md
- 目录速览见 `CODEBASE_MAP.md`；团队口味见 `.claude/project-taste.md`

## 怎么验证
- 全量 lint/测试见各子目录；优先用子目录更快的命令，别动不动跑全量。
```

为什么它好：

- 全文约 20 行，每次会话只占极小一块 context。
- 留下的每一条都过了删除测试——全是「不写 Claude 就会做错」的项目特有信息。
- 模块细节全部下沉到 `services/payments/CLAUDE.md`、`services/risk/CLAUDE.md`，靠指针连过去，靠分层加载按需取用。
- 红线就两条，醒目、不被淹没，Claude 会真的遵守。

**两份文件描述的是同一个项目。差别不在内容多少，在于：精炼版把每一行都用在了「Claude 无从知道、写错会出事」的地方，其余全部交给删除测试、下沉、和指针。**
