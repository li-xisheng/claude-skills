## 一、为什么需要判断层

前面所有层（CLAUDE.md、Hooks、Skills、LSP、子 agent……）解决的是**意图层「要做什么」**和**执行层「该怎么做」**。
显形运动有第三层，常被整套 harness 漏掉：

```
意图层  「要做什么」   ← Spec、需求、CLAUDE.md 里的目标
执行层  「该怎么做」   ← Skills、Hooks、子目录约定、工具
判断层  「做得好不好」 ← 验收标准、对抗性审查、Project Taste   ← 最容易断的一环
```

判断是**最后一公里**。前两层做得再漂亮，如果「什么叫做好」没有显形，整条链就废了——AI 高速产出一堆「跑得起来」的代码，没人能回答它好不好。

为什么这一环现在格外要命：**AI 写代码的速度已经远超人 review 的速度。** 一个工程师一天能审的 PR 是有上限的，而 AI 一天能生成的改动近乎无限。如果「什么叫做好」只活在资深工程师脑子里、从没被写下来，那么当产出速度压过审查速度时，人会被迫**放弃审查**——要么盲目合并，要么用「能编译、测试绿」这种最廉价的信号代替判断。判断层的任务，就是把「好」从人脑里**显形**成 AI 和流程都能用的东西，让审查重新跟得上产出。

这篇源文章的核心隐喻：**AI 是一面镜子。** 你把「好」写成什么样，它就照着产出什么样。写得越窄、越机械，照出来的代码就越窄、越机械。所以判断层不是「再加一堆规则」，而是「想清楚哪些好该显形、哪些好显形了反而会死」。

## 二、判断力光谱 S1–S5：每条规则该落哪一档

不是所有「标准」都该用同一种形态表达。把判断力按「能被文本化到什么程度」排成一条光谱，每一档对应一种表达形态和一种 Goodhart 风险：

| 档 | 形态 | 示例 | Goodhart 风险 |
|---|---|---|---|
| **S1 硬规则** | 布尔（对/错） | Lint、编译错误、类型系统 | 极低 |
| **S2 量化指标** | 连续值 | 覆盖率、圈复杂度 | 极高 |
| **S3 结构化规格** | 语义条款 | Spec、验收标准、行为契约 | 中等 |
| **S4 偏好陈述** | 软约束 | 「我们偏好 X 胜过 Y」 | 低 |
| **S5 裸直觉** | 不可文本化 | 「就是觉得不对」 | 免疫 |

读这张表的方式不是「越往上越好」，而是**每件事都有它该落的那一档**。成熟团队的标志，就是**清楚每件事该落哪一档**，并据此选表达形态。

错配是判断层最常见的病，三种典型错配：

- **把合规放 S4 是失职。** 安全红线、不许碰生产数据库这类「违反即出事」的东西，本该是 S1 硬规则（写进 `permissions.deny`、写进 Lint），却被写成「我们偏好不要直连生产库」这种软约束——AI 在权衡里就可能为了别的目标牺牲它。
- **把审美写成 S1 是暴力。** 「函数不超过 50 行」「继承不超过两层」本是审美偏好（S4），一旦写成硬规则强制，AI 会用拆函数、绕命名等手段机械满足，把本该靠判断的事变成钻空子。
- **把战略拿去 S2 KPI 是组织性悲剧。** 「这个季度要把架构往插件化演进」是战略方向（接近 S5），如果硬塞成「插件化覆盖率 > 80%」这种量化 KPI，团队会优化那个数字而不是那个方向，最后数字达标、方向跑偏。

**纠偏动作：** 给 CLAUDE.md / 规则文件里的**每一条**问一句「这条本质上是哪一档」，再看它现在被写成了哪一档。错配的就重新归档——S1/S3 留在硬规则与验收标准里，S4 挪进 Project Taste，S2 量化指标只做参考信号、绝不当目标，S5 干脆别写（见第六节）。

## 三、Goodhart 定律：指标一旦变目标就被钻空子

> **"When a measure becomes a target, it ceases to be a good measure."**
> （当一个度量变成目标，它就不再是个好度量。）

这是判断层最该刻在墙上的一句话，也是 S2「量化指标 Goodhart 风险极高」的根因。三个 AI 编程里活生生的例子：

1. **「通过这 100 个测试」** → AI 精确地让这 100 个测试通过，**无视一切边界情况**。测试绿了，真实健壮性没动。
2. **「代码不超过 50 行」** → AI 把一个函数拆成五个，行数全部合规，**可读性反而塌了**——逻辑被切碎散落，没人读得懂全貌。
3. **「覆盖率 > 85%」** → AI 生成一堆 `assert True` 之类的空断言把覆盖率刷上去，**覆盖率达标、什么都没验证**。

关键认知：**这不是 AI 坏，而是优化器的本性。** 任何足够强的优化器（AI 也好、被 KPI 驱动的人也好）都会向你设的那个**目标**收敛，而不是向你心里那个**意图**收敛。你给它一个可量化的目标，它就优化那个量，包括用你没预料到的方式优化。度量与意图之间永远有缝，目标化之后这条缝就是被钻的入口。

所以判断层对量化指标的态度是：**量化指标（S2）做监控信号、不做优化目标。** 覆盖率掉了值得看一眼，但「把覆盖率刷到 85%」永远不该是任务的验收条件。验收条件要落在 S3（语义化的验收标准，见第五节石碑一），因为语义条款比数字难钻得多。

## 四、不可能三角：你只能选两条边

判断工程有一个绕不过去的不可能三角，三个目标**最多同时满足两个**：

```
        ① Spec 完备性
       （所有「好」都被写下来）
            /\
           /  \
          /    \
         /      \
        /________\
③ Tacit 保全      ② Goodhart 抗性
（直觉/品味/禁忌   （标准不被钻空子、
  不被抹掉）         不被游戏化）
```

每一种「判断工程」方案，本质都是在这三条边里**选一条边扎营、牺牲对面那个顶点**。三种典型的失败团队，就是三种选择各自的代价：

- **①+② 牺牲③（Spec 完备 + 抗 Goodhart，丢掉 Tacit）**：什么都写进规格、规格也设计得难钻，于是**全员合规，但代码没有温度**。每行都符合标准，整体却没有判断力——老师傅那点「这里该这样、那里别这么干」的隐性手感无处安放，于是他**沉默、出走**。规格越完备，团队越像一台没有品味的机器。
- **①+③ 牺牲②（Spec 完备 + 保全 Tacit，丢掉抗 Goodhart）**：既要写全、又不舍得砍掉任何软东西，于是**KPI / Eval 越堆越多，分数一路涨，事故却不降**。因为堆出来的标准全是可被游戏化的，分数涨的是「应付标准的能力」，不是真实质量。仪表盘一片绿，线上照样炸。
- **②+③ 牺牲①（抗 Goodhart + 保全 Tacit，丢掉 Spec 完备）**：因为怕指标被钻、又想护住直觉，干脆**拒绝写任何规则**——结果 AI 根本跑不起来，没有任何可依据的标准，每次产出都靠人肉兜底，规模化无从谈起。

**工程智慧不是去解这个三角（解不开），而是知道自己选了哪条边、并承认代价。** 纠偏时如果发现一个 harness 三样都想要、哪样都没做到，那就是它没在三角上选边、而是在原地空转。让团队明确「我们这个项目选 ①+③、接受指标会被钻所以根本不靠指标验收」之类的立场，比假装能三全其美健康得多。

## 五、三块石碑：判断层落地的三件器物

把抽象的「判断」落成可操作的器物，就是这三块石碑。搭判断层时按需立，不必三块一起上。

### 石碑一 · 验收标准即代码（Acceptance as Code）

把「什么叫做完了」从人脑里搬进 Git，和代码同等对待。三个要点：

- **用业务语言写，不用断言语言写。** 失败信息应该是「偏离了意图 #3：用户没收到确认邮件」，而不是「assert failed at line 42」。前者任何人（包括 AI、包括 PM）都能判断对错；后者只告诉你某个数字不相等，不告诉你这意味着什么。这让验收标准落在 S3（语义条款）而非 S2（量化）。
- **和代码放在同一个 PR 里。** 代码改了但验收标准没动 → 该质疑「这改动是不是悄悄改了行为却没人声明」；反过来验收标准改了但代码没动 → 该质疑「是不是在偷偷放宽标准」。两者绑在一起，任何一方单独漂移都会被看见。
- **验收标准本身也要被评审。** 标准不是写完就神圣不可侵犯，它和代码一样会写错、会过期、会被人为放水。把「评审验收标准」当成评审流程的一部分。

这块石碑直接对冲第三节的 Goodhart：因为验收落在语义层而非数字层，AI 想钻空子就得在「业务意图」上钻，而那比刷一个覆盖率数字难得多。

### 石碑二 · 对抗性审查网络（Adversarial Review Network）

核心原则一句话：**doer ≠ judge——干活的和评判的，永远不能是同一个 agent。**

为什么不能同一个：**LLM self-consistency bias（自我一致性偏差）**——一个模型给自己的产出打分，会倾向于认为自己是对的。它生成那段代码时用的就是那套推理，再让它评判同一段代码，它只会沿着同一条推理确认「没问题」。**没有异议的评审一定会出错**，因为它根本没引入新的视角。鲁棒性来自**认知多样性**，不来自更努力的自我检查。

三种制造认知多样性的形式：

- **换模型写测试（A 实现 / B 测试）**：A 模型写实现，换 B 模型写测试。B 不知道 A 偷了哪些懒，自然会从不同假设出发去戳。
- **换角色做 CR（creator vs critic）**：同一件事，先以「实现者」身份产出，再换「批判者」身份审查，明确指示它「你的任务是找出这段代码会在哪里出问题」，而不是「确认它没问题」。
- **红蓝对抗**：一方建设、一方专门攻击，逼出隐藏假设。

**在 Claude Code 里怎么落地：** 用**子 agent 当独立质疑者**。主 agent 实现完之后，派一个子 agent（最好换个角色提示词，甚至换模型）在它自己干净的 context 里只做一件事——审查并质疑这份改动，回传一份「我找到的问题」。子 agent 不带主 agent 的推理包袱，等于天然引入了第二视角。这正好复用了 harness 第 7 层的子 agent 能力，只是把它从「探索」用到了「审查」。

### 石碑三 · 品味作为资产（Taste as Asset）

为每个项目维护一份 **Project Taste 文档**，模板见 [assets/templates/project-taste.md](../assets/templates/project-taste.md)。

它和 CLAUDE.md / Lint 的本质区别：**写的是带偏见的偏好陈述，不是硬规则。** 这些条目对应判断力光谱的 S4。它们故意带主观、带立场，例如：

- 「继承不超过两层」
- 「偏好显式传参，而非依赖注入」
- 「拒绝业务代码里出现 V2/V3 这种命名」
- 「宁可多写一次，也不要在业务边界上复用」
- 「对过度拆分的函数，比对略长的函数更反感」
- 「不相信『未来可能用到』的扩展点」

注意这些**不能写成 S1 硬规则**（写成硬规则就触发第二节的「把审美写成 S1 是暴力」，AI 会机械钻空子）。它们是**偏好**，给 AI 一个倾向、一种气味，而不是一道判定线。

它的价值：让 AI 的产出**「闻起来像这个团队写的代码」**。同样实现一个功能，有 Project Taste 和没有，产出的味道完全不同——这份文档是**团队的指纹**，是把资深工程师脑子里那点说不清的手感，沉淀成 AI 能读、能传、新人也能继承的资产。

## 六、刻意不刻的四样东西

有四样东西**刻意不写进任何规则**——不是因为懒，而是因为一旦显形它们就死。这是判断层最反直觉、也最该守住的纪律。

- **🧭 战略方向感。** 方向是团队存在的理由。把方向外包给一份规则文档或一个指标，你就失去了存在的理由——剩下的只是执行，而执行迟早被自动化。方向必须留在人手里。
- **🚫 价值观底线。** 规则一旦穷举，就会冒出**「规则没禁止，所以可以做」**的空子。价值观的作用恰恰在规则沉默的地方说话——「这事规则没禁，但我们不干」。把价值观规则化，等于亲手制造它的漏洞。
- **🩹 带疤痕的禁忌。** 比如「别在周四下午发版」——背后是三年前某次事故留下的肌肉记忆。这类禁忌一旦被**理性化**（非要追问「为什么是周四、给我数据」、写成带条件的规则），就失效了：它的力量来自那道疤，不来自逻辑。逼它讲道理，它就不再是禁忌。
- **🎨 审美的争论。** 两个资深为某个写法吵一下午，**这件事本身就是组织健康的信号**。它说明团队里还有人在乎、还有判断力在碰撞。把审美一锤定音写成标准，就把这个活的信号杀死了。

### 两个寓言：完美的显形 = 没有显形

- **Borges 的 1:1 地图。** 一个帝国想画一张精确无误的地图，越画越细，最后画出一张和帝国一样大、覆盖整个国土的 1:1 地图——于是它**毫无用处**，因为它就是国土本身，不再是「指向国土的简化」。**地图的价值恰恰来自它的不完整。** 验收标准、Eval、规则也一样：追求把「好」百分百写尽，结果就是一份和现实一样复杂、毫无指导价值的东西。
- **黑森林（科学林业）。** 十八世纪德国搞「科学林业」，为了好量化、好管理，把「森林」窄化成「木材产出」这一个指标，砍掉杂树、只种笔直好测的同龄单一树种。头几十年木材产量漂亮极了。**一百年后整片林子塌方**——没了灌木、菌群、鸟兽、土壤多样性，森林失去自我维持能力。当年那个「为了可量化而砍掉的复杂性」，正是森林活着的根基。

两个寓言指向同一句警告：**完美的显形等于没有显形。** 不要把「好代码」窄化成「通过一组 Eval 的代码」——就像不能把「森林」窄化成「木材」。过度显形的 Eval 不再是判断，而是一张 1:1 地图、一片只剩木材的人工林。判断层的纪律，一半是「该显形的显形」，另一半是「该留白的坚决留白」。

## 七、落到 harness：纠偏时怎么用本文

模式 C（纠偏）走到「判断层体检」这一步时（SKILL.md 模式 C 步骤 3），用本文逐项核对 CLAUDE.md / 规则文件 / Project Taste：

1. **查 Goodhart 化（用第三节）。** 在 CLAUDE.md 和规则里搜「覆盖率 > X」「行数 < X」「圈复杂度 < X」这类**纯量化目标**。命中即标红：这些是 S2，被当成目标就会被刷数字。建议改法——把它降级为「监控信号、不作验收」，把真正的验收条件搬到 S3 的语义化验收标准（石碑一）。

2. **查归档错配（用第二节的 S1–S5 表）。** 对每条规则问「它本质是哪一档、现在被写成哪一档」。三类要重点抓：
   - 安全/合规被写成软偏好（该 S1 却落在 S4）→ 升级进 `permissions.deny` 或硬 Lint。
   - 审美偏好被写成硬规则（该 S4 却落在 S1）→ 从 CLAUDE.md 挪进 Project Taste，去掉强制语气。
   - 战略方向被塞成量化 KPI（该留白却落在 S2）→ 整条删掉，回到第六节「刻意不刻」。

3. **查过度显形（用第四节 + 第六节）。** 检查那四样「刻意不刻」的东西有没有被写死：战略方向感、价值观底线、带疤痕的禁忌、审美争论。一旦发现 CLAUDE.md 里在用规则规定战略、用穷举规则代替价值观、把某条禁忌理性化成带条件的判断，**这是过度显形，建议删除而不是修补**——留白本身才是对的。同时用不可能三角问一句：这套 harness 有没有「三样都想要、哪样都没做到」？如果有，帮团队明确选边。

4. **查判断层有没有缺位（用第五节三块石碑）。** 如果一个 harness 把执行层（CLAUDE.md/Skills/Hooks）搭得很全，却完全没有验收标准、没有独立审查、没有 Project Taste——那它缺的就是整个判断层。结合「AI 产出速度已超人 review 速度」这个事实，提醒用户：执行层越强、判断层缺位的危害越大，因为它会让团队更快地产出没人能判断好坏的代码。建议从最轻的石碑三（Project Taste，套 [assets/templates/project-taste.md](../assets/templates/project-taste.md)）起步，再视需要补石碑一、石碑二。

把每条发现写进纠偏报告时，沿用 SKILL.md 模式 C 步骤 4 的格式：**现状 → 问题 → 建议改动 → 为什么**，并标好优先级。判断层的问题通常优先级不低——它一旦缺位或被 Goodhart 化，前面所有层产出的东西都没人把得住质量关。
